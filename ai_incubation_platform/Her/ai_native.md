Agent Native 系统设计方法论
核心主张：从源码抽象出的通用设计原则，适用于任何 Agent 系统架构。

一、代理循环方法论
1.1 最小循环原理

用户输入 → AI 决策 → 工具执行 → 结果反馈 → 继续循环 或 返回响应
方法论：

原则	定义	检查项
循环即代理	Agent = 循环 + 工具 + 状态	是否有持续循环能力？
决策权在 AI	AI 判断何时停止、何时继续	工具是否不含决策逻辑？
工具即能力	工具是 AI 的手脚，不是大脑	工具是否只执行不决策？
状态即记忆	循环依赖状态传递	状态是否跨循环持久？
1.2 循环终止条件

┌─────────────────────────────────────────────────────────────────┐
│                     循环终止决策                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  AI 自主判断：                                                   │
│  ├── 任务完成 → 返回响应                                         │
│  ├── 需要更多信息 → 调用工具                                     │
│  ├── 用户中断 → 停止循环                                         │
│  └── 预算耗尽 → 压缩后继续                                       │
│                                                                 │
│  禁止硬编码终止条件：                                             │
│  ❌ "执行 N 次后停止"                                            │
│  ❌ "调用特定工具后停止"                                          │
│  ✅ AI 根据 stop_reason 自主判断                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

三、权限校验方法论
3.1 多层校验原理

┌─────────────────────────────────────────────────────────────────┐
│                     权限校验多层法则                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Layer 1: 静态规则                                               │
│  ├── 预定义 allow/deny/ask 规则                                  │
│  ├── 基于工具名、参数模式匹配                                     │
│  └── 优点：快速、确定性                                          │
│                                                                 │
│  Layer 2: 动态分类                                               │
│  ├── AI 判断操作安全性                                           │
│  ├── 基于上下文动态决策                                           │
│  └── 优点：灵活、适应复杂场景                                     │
│                                                                 │
│  Layer 3: 用户钩子                                               │
│  ├── 用户自定义校验逻辑                                           │
│  ├── 可阻塞、修改、记录                                           │
│  └── 优点：可扩展、用户控制                                       │
│                                                                 │
│  Layer 4: 环境隔离                                               │
│  ├── 沙箱限制访问范围                                             │
│  ├── 禁止危险操作                                                │
│  └── 优点：兜底安全                                               │
│                                                                 │
│  校验顺序：                                                     │
│  Layer 1 → Layer 2 → Layer 3 → Layer 4 → 执行                   │
│  任一层拒绝 → 停止                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
方法论检查清单：

 是否有静态规则层？
 是否有动态分类层（AI判断）？
 是否有用户钩子层？
 是否有环境隔离层（沙箱）？
 各层是否有明确的拒绝/通过/询问返回？

┌─────────────────────────────────────────────────────────────────┐
│                     Hook 执行法则                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 异步不阻塞原则                                               │
│  ├── Hook 默认异步执行                                           │
│  ├── 主流程继续运行                                               │

│  2. 超时容错原则                                                 │
│  ├── Hook 有超时限制                                             │

│                                                                 │
│  3. 环境传递原则                                                 │
│  ├── Hook 接收完整上下文                                         │
│  ├── 工具名、参数、结果、状态                                     │
│  ├── Hook 可修改上下文                                           │
│                                                                 │
│  4. 输出解析原则                                                 │
│  ├── Hook JSON 输出可解析                                        │
│  ├── 可返回 decision (approve/deny/ask)                         │
│  ├── 可返回 modifiedInput                                        │
│                                                                 │

│  ├── 保留：文件状态缓存、工具结果、会话元数据                      │
│  ├── 压缩边界消息标记历史起点                                     │
│                                                                 │
│  【恢复机制】                                                    │
│  ├── 压缩后自动恢复关键上下文                                     │

│                     Feature Gate 分层法则                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【编译时开关】                                                   │
│  ├── 死代码消除                                                  │
│  ├── 未启用功能不存在于构建产物                                   │
│  ├── 优点：性能最优                                              │
│                                                                 │
│  【运行时开关】                                                   │
│  ├── 动态获取状态                                                │
│  ├── 可实时改变行为                                              │
│  ├── 优点：灵活性                                                │
│                                                                 │
│  【用户级开关】                                                   │
│  ├── 用户可控制                                                  │
│  ├── 设置文件配置                                                │
│                                                                 │
│  【企业级开关】                                                   │
│  ├── 管理员控制                                                  │
│  ├── 用户不可覆盖                                                │
│                                                                 │
│  分层优先级：                                                    │
│  企业级 > 用户级 > 运行时 > 编译时                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
8.2 Kill Switch 设计原理

┌─────────────────────────────────────────────────────────────────┐
│                     Kill Switch 法则                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  设计原则：                                                     │
│  ├── 关键功能必须有 Kill Switch                                  │
│  ├── Kill Switch 可远程触发                                     │
│  ├── Kill Switch 优先级最高                                      │
│                                                                 │
│  应用场景：                                                     │
│  ├── 安全风险 → 立即禁用                                         │
│  ├── 性能问题 → 限流禁用                                         │
│  ├── 成本控制 → 功能限用                                         │
│                                                                 │
│  实现方式：                                                     │
│  ├── 短路返回 → 功能不执行                                       │
│  ├── 降级返回 → 使用备用方案                                     │
│  ├── 阻塞返回 → 强制用户确认                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
九、配置管理方法论
9.1 配置优先级原理

┌─────────────────────────────────────────────────────────────────┐
│                     配置优先级法则                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  优先级层级（高→低）：                                            │
│  1. CLI 参数        → 一次性覆盖                                 │
│  2. 环境变量        → 进程级覆盖                                 │
│  3. 企业策略        → 强制执行                                   │
│  4. 托管配置        → IT 管理                                    │
│  5. 项目配置        → 项目级                                     │
│  6. 用户配置        → 用户级                                     │
│  7. 全局配置        → 默认值                                     │
│  8. 内置默认        → 系统默认                                   │
│                                                                 │
│  冲突解决原则：                                                  │
│  ├── 高优先级覆盖低优先级                                         │
│  ├── 企业策略不可被用户覆盖                                       │
│  ├── CLI 参数优先级最高                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     三层扩展法则                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│                                                                 │

│  ├── 协议标准化                                                  │
│                                                                 │
│  扩展选择原则：                                                  │
│  ├── 行为指导 → Skill                                            │

十一、方法论检查清单
系统设计检查
代理循环：

 是否有持续循环能力？
 AI 是否有决策权（何时停止）？
 工具是否不含决策逻辑？
工具编排：

 是否按只读/写入分类？
 是否有并发安全检测？
 是否正确处理上下文传递？
权限校验：

 是否有多层校验（规则→分类→钩子→隔离）？
 是否有权限模式演进路径？
Hook 系统：

 是否覆盖全生命周期？
 Hook 是否异步不阻塞？
 是否有超时容错？
上下文管理：

 是否有压缩机制？
 是否有预算监控？
 是否有恢复机制？
多代理协作：

 是否有 Agent 分层？
 是否有工具过滤规则？
 是否有隔离机制？
错误恢复：

 是否有错误分层？
 是否有重试策略？
 是否有降级方案？
Feature Gate：

 是否有编译时/运行时开关？
 是否有 Kill Switch？
配置管理：

 是否有优先级层级？
 企业策略是否不可覆盖？
