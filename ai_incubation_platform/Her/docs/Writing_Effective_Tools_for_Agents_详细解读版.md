## 🎯 这篇文章在讲什么？

**一句话总结**：教你怎么给 AI Agent（如 Claude）设计好用的工具，让它更聪明、更高效地完成任务。

**为什么重要**：
- Agent 本身很聪明，但如果工具设计不好，它就像"拿着锤子当螺丝刀用"
- 好的工具能让 Agent 发挥 100% 的能力
- 这篇文章来自 Anthropic 团队的实战经验，不是纸上谈兵

---

## 第一章：什么是"工具"？—— 一个根本性的认知转变

### 1.1 传统软件开发 vs Agent 工具开发

#### 📖 传统软件开发（你熟悉的）

想象你写了一个函数：

```python
def get_weather(city):
    # 调用天气 API，返回结果
    return {"temperature": 25, "condition": "晴"}
```

**特点**：这个函数每次被调用，行为完全一样。
- 输入 `"北京"` → 总是返回北京的天气
- 输入 `"上海"` → 总是返回上海的天气
- **确定性** = 相同输入 → 相同输出

这是**两个确定性系统之间的契约**：调用者和被调用者都按固定规则行事。

---

#### 🤖 Agent 工具开发（完全不同的思维）

现在，你给 Claude 一个天气工具，用户问：**"今天要带伞吗？"**

Claude 会怎么做？它有**多种选择**：

```
用户："今天要带伞吗？"

Claude 可能的选择：
┌─────────────────────────────────────────────────────────────┐
│ 选择 A: 调用天气工具                                          │
│   → 调用 get_weather("用户当前位置")                          │
│   → 根据返回的"降雨概率 30%"判断                              │
│   → 回复："建议带伞，今天有 30% 降雨概率"                      │
├─────────────────────────────────────────────────────────────┤
│ 选择 B: 用通用知识回答                                        │
│   → 不调用工具                                               │
│   → 回复："我不知道你在哪个城市，请告诉我你的位置"              │
├─────────────────────────────────────────────────────────────┤
│ 选择 C: 先问澄清问题                                          │
│   → 回复："你在哪个城市？这样我可以查询当地天气"               │
│   → 等用户回复后，再调用工具                                   │
├─────────────────────────────────────────────────────────────┤
│ 选择 D: 理解错误（偶尔发生）                                   │
│   → 误解工具用法，调用失败                                     │
│   → 或者"幻觉"，编造一个天气数据                               │
└─────────────────────────────────────────────────────────────┘
```

**特点**：Agent 是**非确定性系统**
- 相同的问题，Agent 可能选择不同的策略
- 它需要"思考"才能决定怎么做
- 有时它会犯错（误解工具、幻觉）

这是**确定性系统（工具）与非确定性系统（Agent）之间的契约**。

---

### 1.2 核心认知转变

#### ❌ 错误思维

```
"我就把现有的 API 封装一下，给 Agent 用就行"
```

这就像：你有一堆螺丝刀、锤子、扳手（传统 API），然后直接扔给一个新手（Agent），说"你自己想办法用"。

**问题**：Agent 不知道什么时候用哪个、怎么组合、怎么处理异常。

---

#### ✅ 正确思维

```
"我需要专门为 Agent 设计工具，考虑它的认知特点"
```

这就像：你观察新手的工作习惯，然后为他设计一套"智能工具箱"——每个工具都有清晰的标签、说明书，甚至组合工具让他一步完成常见任务。

---

### 1.3 一个形象的类比

| 维度 | 传统软件开发 | Agent 工具开发 |
|------|-------------|---------------|
| **契约对象** | 稳定的程序 → 稳定的程序 | 稳定的程序 → 需要思考的 Agent |
| **类比** | 给机器人写指令（精确执行） | 给新员工写说明书（需要理解） |
| **错误处理** | 程序崩溃 → 报错堆栈 | Agent 困惑 → 可能做错事或"幻觉" |
| **设计重点** | 功能完整、性能优化 | 易理解、易使用、减少认知负担 |

## 第二章：怎么开始？—— 四步流程

### 流程图（像产品迭代一样）

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   Step 1: 快速原型                                              │
│   └───────────────                                              │
│   别想太多，先写个能用版本                                        │
│   自己测试，看看哪里不顺手                                        │
│                                                                 │
│                              ↓                                  │
│                                                                 │
│   Step 2: 创建评估                                              │
│   ───────────────                                              │
│   设计一堆测试任务（真实的、复杂的）                               │
│   用程序跑这些任务，收集数据                                       │
│                                                                 │
│                              ↓                                  │
│                                                                 │
│   Step 3: 分析 + Agent 协作                                      │
│   ─────────────────────                                         │
│   让 Claude 分析测试结果                                         │
│   让 Claude 帮你改进工具                                         │
│                                                                 │
│                              ↓                                  │
│                                                                 │
│   Step 4: 循环迭代                                              │
│   ────────────────                                              │
│   改完再测，测完再改                                             │
│   直到性能达标                                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

### Step 1: 快速原型

#### 怎么做？

1. **别过度设计**：先写个能用的版本，哪怕不完美
2. **给 Claude 文档**：如果你用的是某个库/API，把文档给它（最好找 `llms.txt` 这种 LM-friendly 格式）
3. **封装成 MCP Server**：这样可以在 Claude Code 或 Claude Desktop 里直接测试

#### 连接方式

```bash
# Claude Code 中添加本地 MCP Server
claude mcp add my_tools python server.py

# Claude Desktop: Settings > Developer
```

4. **自己测试**：像用户一样用，感受哪里别扭

---

### Step 2: 创建评估（这是最关键的）

#### 什么是"评估"？

就是给 Agent 一堆任务，看它能不能完成，记录数据。

**类比**：给学生出考试题，看他能答对多少，分析哪里错了。

---

#### 什么样的任务是好任务？

##### ✅ 强任务（推荐）

特点：**真实、复杂、需要多步操作**

**示例 1**：
```
下周安排与 Jane 的会议讨论 Acme Corp 项目。
要求：
1. 附上上次项目规划会议的笔记
2. 预定会议室
3. 找一个双方都有空的时间
```

**为什么强？**
- 需要 3 个以上工具调用（查日历、找笔记、预定会议室）
- 模拟真实工作场景
- Agent 必须理解任务、分解步骤、组合工具

---

**示例 2**：
```
客户 ID 9182 报告单次购买被扣款三次。
请：
1. 查找所有相关日志条目
2. 确定是否有其他客户受影响
3. 总结问题原因
```

---

##### ❌ 弱任务（不推荐）

特点：**过于简单、只测单一功能**

**示例**：
```
下周与 jane@acme.corp 安排会议。
```

**为什么弱？**
- 没有复杂度，一个工具调用就完成
- 不能测试 Agent 的推理能力
- 不能暴露工具设计的问题

---

#### 怎么验证答案？

| 方法 | 优点 | 缺点 |
|------|------|------|
| **字符串匹配** | 简单、快速 | 太严格，格式不同就判错 |
| **Claude 判断** | 灵活、智能 | 成本高、可能主观 |
| **混合方案** | 平衡 | 推荐 |

**重要**：不要因为格式、标点、措辞不同就判错。只要答案实质正确，就应该通过。

---

#### 怎么运行评估？

用程序跑，不是人工测试：

```python
# 简化的评估循环
for task in evaluation_tasks:
    while not task_complete:
        # 1. 让 Agent 看任务
        response = call_llm(task_prompt)
        
        # 2. 如果 Agent 要调用工具，执行工具
        if response.has_tool_call:
            tool_result = execute_tool(response.tool_call)
            # 把工具结果返回给 Agent
        
        # 3. Agent 继续思考，直到给出答案
        task_complete = response.is_final
```

---

#### 要收集什么数据？

| 数据 | 用途 |
|------|------|
| **成功率** | 基础指标 |
| **总运行时间** | 性能分析 |
| **工具调用次数** | 是否有冗余调用？ |
| **Token 消耗** | 是否浪费 context？ |
| **错误次数** | 参数问题？描述不清？ |

---

### Step 3: 分析结果 + Agent 协作

#### 怎么分析？

1. **看 Agent 困惑的地方**
   - 哪些任务经常失败？
   - 哪些步骤 Agent 会卡住？

2. **读 Agent 的"思考过程"**
   - 让评估 Agent 输出 reasoning/feedback
   - 看 Agent 自己怎么解释失败原因
   - **但注意**：Agent 不总是说实话，要看原始 transcript

3. **看原始 transcript**
   - Agent 调了哪些工具？
   - 工具返回了什么？
   - Agent 怎么处理返回结果的？

---

#### 一个真实案例

**Claude Web Search 工具的故事**：

```
问题发现：
┌─────────────────────────────────────────────────────────────┐
│ 用户搜索："最新的 AI 新闻"                                    │
│                                                              │
│ Claude 调用：search_web(query="最新的 AI 新闻 2025")         │
│                                                              │
│ 问题：Claude 无缘无故给 query 加了 "2025"                     │
│      → 搜索结果偏向 2025 年的内容                             │
│      → 可能遗漏 2026 年的重要新闻                             │
└─────────────────────────────────────────────────────────────┘

根因分析：
┌─────────────────────────────────────────────────────────────┐
│ Claude "以为"加上年份会得到更准确的结果                        │
│ 但工具描述没有明确说明"不要加年份"                             │
└─────────────────────────────────────────────────────────────┘

修复：
┌─────────────────────────────────────────────────────────────┐
│ 改进工具描述：                                                │
│ "query 参数：搜索关键词，不要包含年份，搜索引擎会自动           │
│  返回最新结果"                                                │
│                                                              │
│ 效果：问题消失                                                │
└─────────────────────────────────────────────────────────────┘
```

---

#### 让 Claude 帮你改进工具

**方法**：
1. 把所有评估 transcript 拼成一个文件
2. 粘贴到 Claude Code
3. 说："分析这些 transcript，找出工具的问题，给出改进方案"

**效果**：
- Claude 能发现你没注意到的问题
- Claude 能批量重构多个工具
- Anthropic 团队用这个方法写了这篇文章的大部分建议

## 第三章：工具设计的五大原则（核心干货）

### 原则一：选择正确的工具（少即是多）

#### ❌ 最常见错误

```
"我有很多 API，我把它们全部封装成工具给 Agent 用"
```

**问题**：
- 工具太多 → Agent 不知道选哪个
- 工具重叠 → Agent 会困惑
- Agent 上下文有限 → 加载太多工具描述会占满 context

---

#### 📖 核心洞察：Agent 的"上下文"是稀缺资源

这是一个关键认知转变：

| 维度 | 计算机程序 | AI Agent |
|------|-----------|----------|
| **内存** | 几十 GB，便宜充足 | 几万 tokens，昂贵稀缺 |
| **处理方式** | 可以遍历所有数据 | 逐 token 读取，读到哪用到哪 |

**形象的比喻**：

```
计算机程序：
┌─────────────────────────────────────────────────────────────┐
│ 想象你在图书馆找一本书                                        │
│                                                              │
│ 计算机可以：                                                  │
│   1. 把整个目录读一遍（几千条记录）                            │
│   2. 一条一条比对                                             │
│   3. 找到匹配的书                                             │
│                                                              │
│ 这没问题，因为计算机内存大，处理快                              │
└─────────────────────────────────────────────────────────────┘

AI Agent：
┌─────────────────────────────────────────────────────────────┐
│ 想象你在图书馆找一本书                                        │
│                                                              │
│ Agent 的困境：                                                │
│   1. 只能"看"几页内容（上下文有限）                            │
│   2. 如果工具返回整个目录（几千条）                             │
│   3. Agent 要逐条读取 → 读几条上下文就满了                     │
│   4. 还没找到目标，就"看不下去了"                              │
│                                                              │
│ 这就像让你在图书馆从 A 到 Z 逐页翻目录找一本书                  │
│ 你会崩溃的！                                                  │
└─────────────────────────────────────────────────────────────┘
```

---

#### ✅ 正确做法：设计"智能跳过"的工具

**例子：地址簿搜索**

| 设计 | 工具 | Agent 体验 |
|------|------|-----------|
| ❌ 传统思维 | `list_contacts()` → 返回所有联系人 | Agent 要逐条读取，上下文爆炸 |
| ✅ Agent 思维 | `search_contacts(name="张")` → 只返回匹配的 | Agent 直接看到结果，高效 |

**类比**：
- ❌ `list_contacts` = 让你从 A 到 Z 翻整个通讯录
- ✅ `search_contacts` = 让你直接跳到"张"字开头的页面

---

#### 📖 工具合并策略

**核心思想**：把多个相关操作合并成一个工具

| 分散设计（不推荐） | 合并设计（推荐） |
|------------------|----------------|
| `list_users` | |
| `list_events` | `schedule_event` |
| `create_event` | （一次调用：查可用时间 + 预定会议） |
| | |
| `read_logs` | `search_logs` |
| | （只返回相关日志 + 周几行上下文） |
| | |
| `get_customer_by_id` | `get_customer_context` |
| `list_transactions` | （一次返回：客户信息 + 交易记录 + 备注） |
| `list_notes` | |

**好处**：
1. 减少 Agent 需要调用的工具数量
2. 减少"中间结果"消耗的上下文
3. 让 Agent 像"人类高手"一样工作，而不是机械地分步操作

---

#### 工具边界原则

- 每个工具有**一个明确的目的**
- 工具名应该让 Agent **一眼知道干什么**
- 不做重叠功能的工具

---

### 原则二：命名空间（给工具分组）

#### 问题场景

```
Agent 可能有：
- 50 个 MCP servers
- 200+ 个工具

工具重叠时会困惑：
┌─────────────────────────────────────────────────────────────┐
│ 工具 A: search()      ← Jira 的搜索                          │
│ 工具 B: search()      ← Slack 的搜索                         │
│ 工具 C: search()      ← Asana 的搜索                         │
│                                                              │
│ Agent："我要搜索项目信息，该调用哪个 search()？"               │
│        → 可能选错                                            │
│        → 任务失败                                            │
└─────────────────────────────────────────────────────────────┘
```

---

#### ✅ 解决方案：命名空间

**按服务分组**：
```
jira_search
slack_search  
asana_search
```

**按资源分组**：
```
asana_projects_search
asana_users_search
asana_tasks_search
```

**效果**：
- Agent 看到 `asana_projects_search`，立刻知道：
  - 这是 Asana 的工具
  - 用来搜索项目
- 不需要猜测

---

#### 实验发现

Anthropic 团队测试发现：
- `prefix_search`（如 `asana_search`）vs `search_suffix`（如 `search_asana`）
- 对评估结果有**显著影响**
- 但哪个更好，取决于具体的 LLM
- **建议**：你自己测试，选最适合你场景的方案

---

### 原则三：返回有意义的上下文

#### ❌ 常见错误：返回一堆技术标识符

```json
{
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "avatar_url_256px": "https://cdn.example.com/avatars/256/a1b2...",
  "mime_type": "image/png",
  "created_at_timestamp": 1704067200
}
```

**问题**：
- UUID 对 Agent 是无意义的字符串
- Agent 很难记住、引用 UUID
- 容易产生幻觉（记错 UUID）

---

#### ✅ 正确做法：返回高信号信息

```json
{
  "name": "张三",
  "avatar_url": "https://cdn.example.com/avatars/a1b2...",
  "file_type": "PNG 图片",
  "created_at": "2024年1月1日"
}
```

**好处**：
- Agent 能理解这些信息
- 更容易记住、引用
- 减少幻觉

---

#### 📖 一个重要发现

Anthropic 团队发现：

```
将 UUID 转换为语义上有意义的标识符：
┌─────────────────────────────────────────────────────────────┐
│ 原方案：用 UUID                                               │
│   user_id: "a1b2c3d4-e5f6..."                                │
│                                                              │
│   Agent 检索精度：65%                                         │
│   幻觉率：15%                                                 │
├─────────────────────────────────────────────────────────────┤
│ 改进方案：用 0-indexed ID                                     │
│   user_id: "user_001"                                        │
│                                                              │
│   Agent 检索精度：92%                                         │
│   幻觉率：3%                                                  │
└─────────────────────────────────────────────────────────────┘

结论：即使是简单的数字编号，也比 UUID 强得多！
```

---

#### 灵活性方案：支持两种格式

有时 Agent 需要 ID 来调用下游工具，有时只需要内容。

**方案**：加一个 `response_format` 参数

```python
def get_slack_thread(thread_ts, response_format="concise"):
    if response_format == "detailed":
        return {
            "thread_ts": "1234567890.123456",  # 用于后续操作
            "channel_id": "C12345678",
            "user_id": "U12345678",
            "content": "讨论内容..."
        }
    else:  # concise
        return {
            "content": "讨论内容..."
            # 不包含 ID，节省 token
        }
```

**效果**：
- Detailed：206 tokens（用于后续 tool call）
- Concise：72 tokens（约 1/3，用于简单展示）

### 原则四：Token 效率优化

#### 为什么重要？

Agent 的上下文有限（如 Claude Code 默认 25,000 tokens）
- 工具返回太多内容 → 占满上下文
- Agent 无法继续思考 → 任务失败

---

#### 四种优化方法

| 方法 | 说明 | 何时使用 |
|------|------|---------|
| **Pagination** | 分页返回，每页固定数量 | 数据量大时 |
| **Range selection** | 让 Agent 指定范围 | 时间范围、ID 范围等 |
| **Filtering** | 让 Agent 指定过滤条件 | 搜索场景 |
| **Truncation** | 超过限制截断 + 提示 | 默认保护 |

---

#### 📖 Truncation 最佳实践

**错误做法**：
```
结果被截断。请缩小搜索范围。
```

Agent 可能不知道怎么做。

---

**正确做法**：
```
结果已截断（只显示前 100 条）。如需更多结果，请使用：
- 过滤参数：query="关键词"
- 时间范围：time_range="last_hour"
- 分页参数：page=2, page_size=20

示例：search_logs(query="error", time_range="last_hour")
```

**效果**：Agent 知道下一步该怎么做。

---

#### 📖 错误响应最佳实践

**无帮助的错误**：
```
Error: Invalid parameter
```

Agent 不知道哪个参数错了、怎么修正。

---

**有帮助的错误**：
```
Error: 'date' 参数格式错误。
预期格式：ISO 8601（如 '2025-01-15' 或 '2025-01-15T10:30:00'）
你传入的值：'Jan 15'

建议：改用 '2025-01-15'
```

**效果**：Agent 能根据提示修正参数，继续完成任务。

---

### 原则五：Prompt-engineering 工具描述

#### 为什么这是"最有效方法"？

工具描述会加载到 Agent 的 context 中
- 清晰的描述 → Agent 正确使用
- 模糊的描述 → Agent 猜测、犯错

---

#### 📖 编写原则

**思维模型**：像向新员工介绍工具一样

```
新员工入职，你教他用内部系统：

❌ 错误介绍：
"这个 search 工具可以搜索东西"

✅ 正确介绍：
"这个 search_projects 工具可以搜索项目。
参数说明：
- query: 搜索关键词，支持项目名称、描述中的文字匹配
- status: 可选，筛选项目状态（active/completed/archived）
- owner_id: 可选，筛选项目负责人

使用场景：
- 当用户想查找某个项目时调用
- 当用户问'我的项目有哪些'时，用 owner_id=当前用户ID

注意事项：
- query 不要加年份，系统会自动返回最新结果
- 如果结果太多，加 status 过滤会更精确"
```

---

#### 参数命名的学问

| 命名 | 问题 |
|------|------|
| ❌ `user` | Agent 不知道是 user_id、user_name 还是 user_email |
| ✅ `user_id` | 明确是 ID |
| ✅ `user_name` | 明确是姓名 |

## 附录：术语表

| 术语 | 解释 |
|------|------|
| **确定性系统** | 相同输入总是产生相同输出的系统（如传统程序） |
| **非确定性系统** | 相同输入可能产生不同输出的系统（如 Agent） |
| **上下文 (Context)** | Agent 能"看到"的信息量，以 token 为单位，有限且昂贵 |
| **Token** | 文本的最小单位，约 4 个字符 = 1 token（中文约 1-2 字 = 1 token） |
| **MCP (Model Context Protocol)** | Anthropic 推动的工具协议，让 Agent 能调用外部工具 |
| **Transcript** | Agent 执行任务的完整记录（思考、工具调用、返回结果） |
| **幻觉 (Hallucination)** | Agent 编造不存在的信息，如记错 UUID |
| **命名空间** | 给工具名加前缀分组，如 `asana_search` |
| **Pagination** | 分页返回数据，避免一次返回太多 |
| **Truncation** | 超过限制截断数据，并给出提示 |
| **Prompt-engineering** | 优化提示词/描述，让模型更好地理解和使用 |
