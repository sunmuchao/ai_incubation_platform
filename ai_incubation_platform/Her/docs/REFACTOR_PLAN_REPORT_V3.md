## 三、核心架构问题（五问法根因分析）

### 3.1 测试收集错误问题（P0 - 最高优先级）

**问题现象**: pytest 收集测试时有 8 个错误

```
问题现象：测试收集阶段出现 8 个错误，影响测试执行
├─ 为什么 1: 部分测试文件存在导入或语法问题
├─ 为什么 2: 模型迁移后导入路径未完全更新
├─ 为什么 3: 测试文件未同步重构
├─ 为什么 4: 缺乏测试文件的自动化同步机制
└─ 为什么 5: **根因：重构时未执行全量测试验证**

根本对策：
1. 定位并修复 8 个测试收集错误
2. 建立重构后自动运行测试的流程约束
3. 添加 pre-commit hook 强制测试通过
```

**行动项**:
- 执行 `pytest tests/ --collect-only -v` 定位具体错误
- 逐一修复导入路径或语法问题
- 运行全量测试确保 100% 通过

### 3.2 API 路由层语义化命名待实施（P1）

**问题现象**: `docs/api-renaming-plan.md` 中 API 路由层重构仍未执行

```
问题现象：API 路由仍使用 P11/P12 等编号命名
├─ 为什么 1: 模型层重构已完成，路由层重构推迟
├─ 为什么 2: API 路由变更涉及前端调用，影响范围大
├─ 为什么 3: 缺乏 API 版本管理策略
├─ 为什么 4: 担心破坏现有功能

根本对策：
1. 采用渐进式迁移：旧路由保留 + 新路由添加
2. 使用路由别名过渡期

### 3.3 服务层膨胀问题（P2）

**问题现象**: services 目录有 92 个文件，职责边界模糊

```
问题现象：服务层文件过多，部分职责重叠
├─ 为什么 1: 每个功能点都创建独立服务
├─ 为什么 2: 缺乏服务职责边界规范
├─ 为什么 3: 未评估服务合并可能性
├─ 为什么 4: 功能迭代各自添加服务
└─ 为什么 5: **根因：缺乏服务层架构治理**

根本对策：
1. 按功能域合并相似服务（如 ai_* 相关服务）

3. 评估 BaseService 利用率，统一 CRUD 模式
