### 四、核心算法

#### 4.1 双向选择概率

```
P(双向匹配) = P(A喜欢B) × P(B喜欢A) × 竞争分惩罚

竞争分惩罚：
├── 分差 <10分 → ×1.0（无惩罚）
├── 分差 10-20分 → ×0.9
├── 分差 20-30分 → ×0.75
└── 分差 >30分 → ×0.6
```

#### 4.2 匹配分层策略

```
层级差距惩罚：
├── 同层 → ×1.0（无惩罚）
├── 跨1层 → ×0.85
├── 跨2层 → ×0.6
├── 跨3层 → ×0.35
└── 跨4层+ → ×0.15

弥补机制：匹配度高可弥补最多50%惩罚
```

## 一、背景与问题

### 1.1 当前匹配系统的问题

现有匹配系统基于"相似度"假设：两个用户越相似，匹配度越高。这种假设存在根本性缺陷：

```
问题1：相似不等于匹配
├── 两个强势的人 → 相似但会冲突
├── 两个内向的人 → 相似但可能没话聊
└── 焦虑型 × 回避型 → 可能相似但最差组合

问题2：忽略了互补匹配
├── 内向 + 外向 → 可能是完美互补
├── 理性 + 感性 → 可能互补决策
└── 强势 + 温和 → 可能相处和谐

问题3：规则僵化
├── 无法从数据中学习
├── 无法适应不同用户的偏好
└── 无法持续优化
```

### 1.2 多学科匹配理论

根据心理学、社会学、生物学等多学科研究，匹配是多维度的：

```
┌─────────────────────────────────────────────────────────────────────────┐
│  维度分类                                                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【必须相似】                                                            │
│  ├── 价值观（人生目标、婚恋观、家庭观）                                  │
│  ├── 依恋类型（安全型最佳）                                              │
│  └── 消费观念                                                            │
│                                                                         │
│  【相似更好】                                                            │
│  ├── 教育背景                                                            │
│  ├── 生活节奏                                                            │
│  └── 沟通风格                                                            │
│                                                                         │
│  【可以互补】                                                            │
│  ├── 性格（内向+外向）                                                   │
│  ├── 技能（你会的我不会）                                                │
│  └── 社交风格                                                            │
│                                                                         │
│  【不能太相似】                                                           │
│  ├── 强势程度（两个都强势会冲突）                                        │
│  └── 控制欲（都太强会互相折磨）                                          │
│                                                                         │
│  【有交集即可】                                                           │
│  ├── 兴趣爱好                                                            │
│  └── 话题偏好                                                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 二、核心思想

### 2.1 范式转变

```
传统匹配：相似度 = distance(A, B) 越小越好
─────────────────────────────────────────────
问题：假设"越近越好"，忽略互补匹配

新匹配：  匹配 = B ∈ f(A)
─────────────────────────────────────────────
含义：B 是否在 A 的匹配区域内
优势：不假设"越近越好"，支持相似和互补
```

### 2.2 核心概念

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  用户向量：                                                              │
│  将用户映射到 N 维向量空间中的一个点                                     │
│  用户 = [v1, v2, v3, ..., vN]                                          │
│                                                                         │
│  匹配区域：                                                              │
│  一个或多个"球体"区域，区域内的人都可能匹配                              │
│  区域 = {center, radius, weight, type}                                  │
│                                                                         │
│  匹配函数：                                                              │
│  f(用户向量) → [匹配区域1, 匹配区域2, ...]                              │
│  由 LLM 从成功/失败案例中学习得到                                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 图解对比

```
旧方案：找相似的人

              用户A
               ●
              ╱│╲
             ╱ │ ╲
            ╱  │  ╲
           ●   ●   ●  ← 只找附近的人
           
           
新方案：找匹配区域的人

              用户A
               ●
               
          ●           ← 匹配区域1（互补型）
          
          
               ●      ← 匹配区域2（相似型）
               
               
                    ● ← 匹配区域3（混合型）
                    
这些区域可能离用户A很远，但里面的人才是适合的！
```

```python
# 价值观一票否决规则
VALUE_VETO_RULES = [
    {
        "dimension": "v17",  # 是否想要孩子
        "dimension_idx": 17,
        "veto_threshold": 0.5,  # 差异超过0.5触发
        "veto_action": "score_penalty",  # 直接扣减50%匹配分
        "penalty_factor": 0.5,
        "reason": "生育意愿分歧是根本性冲突，无法调和"
    },
    {
        "dimension": "v27",  # 消费观念
        "dimension_idx": 27,
        "veto_threshold": 0.7,  # 差异超过0.7触发
        "veto_action": "score_penalty",
        "penalty_factor": 0.4,
        "reason": "金钱观分歧会导致持续的日常冲突"
    }
]

def apply_value_veto(user_a_vector, user_b_vector, base_score):
    """应用价值观硬性约束"""
    for rule in VALUE_VETO_RULES:
        dim_idx = rule["dimension_idx"]
        diff = abs(user_a_vector[dim_idx] - user_b_vector[dim_idx])
        
        if diff > rule["veto_threshold"]:
            base_score *= rule["penalty_factor"]
            # 记录否决原因
            logger.info(f"Value veto triggered: {rule['reason']}")
    
    return base_score
```

#### 4.2.4 依恋类型维度 [48-63]

> **设计说明**：依恋类型在实际心理学观测中具有**动态流动性**。一个"焦虑型"用户在遇到极其稳定的"安全型"伴侣后，可能会表现出安全倾向；反之，两个"焦虑型"相遇会触发严重的防御机制。此外，人在热恋期（数据采集初期）往往都会伪装成"安全型"，只有在冲突（行为数据）发生后，真实的依恋底色才会显露。因此，依恋维度的更新频率调整为"中"，并引入"压力状态下的依恋表现"子维度。
>
> **依恋互补因子**：匹配函数不应只看当前得分，更应加入"依恋互补因子"。**主动增加"焦虑型"与"安全型"的匹配权重**，因为安全型伴侣能帮助焦虑型逐步向安全型转化，这是正向的"治愈关系"。而不是简单地排除回避型。
>
> **安全型减震器效应**："焦虑型"与"回避型"的组合是极其痛苦的"追逃模式"。虽然黑名单规则会排除回避型，但更重要的策略是：**将"安全型得分"(v48) 作为所有匹配区域的权重加成项**。安全型的人像关系的"减震器"，能与任何类型的人建立更稳定的关系。无论对方是焦虑型、回避型还是恐惧型，安全型伴侣都能提供稳定的情感支持。

```
维度 48-51: 依恋类型编码（基础）
├── v48: 安全型得分
├── v49: 焦虑型得分
├── v50: 回避型得分
└── v51: 恐惧型得分

维度 52-55: 关系模式
├── v52: 亲密需求程度
├── v53: 独立需求程度
├── v54: 安全感需求
└── v55: 承诺意愿

维度 56-63: 压力状态下的依恋表现（动态）
├── v56: 冲突时的情绪反应模式（冷静/激动/回避）
├── v57: 冲突时的沟通倾向（主动解决/被动等待/逃避）
├── v58: 被拒绝时的反应（接受/焦虑追问/冷漠）
├── v59: 伴侣忙碌时的反应（独立等待/焦虑需求/抱怨）
```

```python
SECURE_ATTACHMENT_DAMPENER = {

    "apply_to": "all_match_regions",  # 应用到所有匹配区域

# 焦虑型 + 回避型：恶性循环组合（一票否决）
# 已在黑名单区域中定义

def apply_secure_dampener(user_vector: np.ndarray, base_score: float) -> float:
    """
    应用安全型减震器效应
    安全型得分(v48)越高，匹配分越高，无论对方是什么依恋类型
    """
    secure_score = user_vector[48]  # 安全型得分
    bonus = min(secure_score * 0.2, 0.2)  # 最高增加20%
    return base_score * (1 + bonus)
```

**"双冷模式"预警规则**：

```python
# 双冷模式预警：双方都是冷战倾向者
COLD_MODE_WARNING = {
    "trigger_conditions": [
        {"dimension": "v140", "operator": "==", "value": "cold_war"},  # 双方都是冷战型
        {"dimension": "v141", "operator": ">", "value": 0.6},  # 双方冷战倾向都高
    ],
    "action": "warning",  # 不是一票否决，而是预警
    "warning_message": "你们都倾向于冷战回避，一旦产生矛盾可能难以修复。建议你们在学习如何主动沟通后再尝试匹配。",
    "score_penalty": 0.3,  # 降低匹配分30%
    "alternative_recommendation": "建议匹配一个冲突修复能力强、主动沟通型的伴侣"
}

def check_cold_mode(user_a_vector, user_b_vector):
    """检查是否为双冷模式"""
    a_cold = user_a_vector[141] > 0.6  # 冷战倾向
    b_cold = user_b_vector[141] > 0.6
    
    a_avoid = user_a_vector[140] in ["cold_war", "avoid"]  # 冲突沟通方式
    b_avoid = user_b_vector[140] in ["cold_war", "avoid"]
    
    if a_cold and b_cold and a_avoid and b_avoid:
        return True, COLD_MODE_WARNING["warning_message"]
    return False, None
```

#### 4.2.10 隐性特征维度 [144-151]

> **伦理与准确性说明**：从"说的"和"做的"差异中推断隐性偏好（例如嘴上说喜欢安静，实际喜欢活跃型）存在"社会期许偏差"风险。用户可能因为自卑或外界压力而否定自己的真实渴望。如果 LLM 强行将其归类为"口是心非"，可能会引发用户的反感，甚至造成"信息茧房"，让用户困在自己潜意识的欲望里无法成长。
>
> **权重策略**：隐性特征向量在初次匹配时应**保持克制**，权重设置为 0.1-0.2，更多用于后期复盘和推荐理由的微调，而非主导匹配决策。
>
> **LLM学习重点**：隐性特征挖掘应作为 LLMLearningService 的核心任务。让 LLM **重点分析那些"实际行为背离注册标签"的成功案例**——这往往是算法实现"比你更懂你"的关键突破口。例如：用户声明喜欢安静内向型，但实际滑动行为显示偏好外向活跃型，且与外向型用户匹配后关系持续良好。这种案例揭示了用户的潜意识需求，是比问卷更真实的匹配依据。

```
维度 144-151: 隐性偏好
├── v144: 声明偏好 vs 实际行为差异度（整体）
├── v145: 实际偏好的性格类型（从行为推断）
├── v146: 实际偏好的生活方式类型
├── v147: 实际偏好的沟通风格类型
├── v148: 潜意识需求推断（可能未意识到的）
├── v149: 社会期许偏差修正系数
├── v150: 隐性特征置信度（行为样本量 vs 声明一致性）
```

#### 4.2.13 市场吸引力维度 [168-175]

> **设计背景**：现实中，"白富美/高富帅"现象是客观存在的。大部分男性偏好年轻貌美，大部分女性偏好高收入、身材高大的男性。如果系统只推"用户想要的"而不考虑"对方是否可能接受"，会导致大量无效匹配——用户喜欢对方，但对方看不上用户。
>
> **核心原则**：**不是推"你想要的"，而是推"你想要的 + 也想要你的"**。系统内部计算"市场竞争分"和"双向选择概率"，但不展示给用户，避免物化用户和引发反感。

```

├── v168: 被喜欢率（被like次数 / 曝光次数）
├── v169: 主动喜欢转化率（发出的like / 浏览次数）
├── v170: 收到消息率（收到消息数 / 匹配数）
├── v171: 回复率（回复消息数 / 收到消息数）
├── v172: 竞争分（综合市场价值评估，0-100）
├── v173: 偏好目标竞争分（用户喜欢的类型的市场分）
├── v174: 双向匹配概率系数（与目标类型互选的概率）
└── v175: 市场吸引力置信度（行为样本量是否足够）
```

**市场竞争分计算逻辑**：

```python
class MarketDynamicsService:
    """市场竞争动态分析服务"""
    
    def compute_competition_score(
        self, 
        user_vector: np.ndarray,
        behavior_data: dict
    ) -> float:
        """
        计算用户的"市场竞争分"（0-100，内部使用，不展示）
        
        基于客观条件 + 市场反馈：
        - 男性：身高、收入、房车、学历（权重高）
        - 女性：年龄、外貌（通过被喜欢率间接衡量）、学历（权重中等）
        
        注意：这个分数永不展示给用户，只用于内部匹配优化
        """
        gender = user_vector[3]  # 性别
        
        if gender == 0:  # 男性
            # 男性市场竞争分权重
            score = (
                user_vector[11] * 30 +      # 身高权重30%
                user_vector[12] * 35 +      # 收入权重35%
                user_vector[14] * 10 +      # 房产权重10%
                user_vector[15] * 5 +       # 车权重5%
                user_vector[10] * 10 +      # 学历权重10%
                behavior_data.get("被喜欢率", 0.5) * 10  # 市场反馈权重10%
            ) * 100
        else:  # 女性
            # 女性市场竞争分权重
            score = (
                (1 - user_vector[0]) * 35 +  # 年龄（年轻=高分）权重35%
                behavior_data.get("被喜欢率", 0.5) * 30 +  # 市场反馈（外貌）权重30%
                user_vector[10] * 15 +       # 学历权重15%
                user_vector[11] * 5 +        # 身高权重5%
                user_vector[12] * 10 +       # 收入权重10%
                user_vector[14] * 5          # 房产权重5%
            ) * 100
        
        return min(score, 100)  # 最高100分
    
    def compute_mutual_match_probability(
        self,
        user_a_vector: np.ndarray,
        user_b_vector: np.ndarray,
        a_competition_score: float,
        b_competition_score: float
    ) -> float:
        """
        计算双向选择概率
        
        核心思想：只有双方都可能互相接受时，才是有效匹配
        
        Returns:
            双向匹配概率（0-1）
        """
        # A对B的单向偏好概率
        a_likes_b = self._compute_单向偏好(user_a_vector, user_b_vector)
        
        # B对A的单向偏好概率
        b_likes_a = self._compute_单向偏好(user_b_vector, user_a_vector)
        
        # 竞争分差异惩罚（分差太大，双向概率降低）
        score_diff = abs(a_competition_score - b_competition_score)
        score_penalty = max(0, 1 - score_diff / 30)  # 分差30分以上，惩罚到0
        
        # 双向匹配概率 = 两者乘积 × 竞争分惩罚
        mutual_prob = a_likes_b * b_likes_a * score_penalty
        
        return mutual_prob
    
    def _compute_单向偏好(
        self,
        viewer_vector: np.ndarray,
        target_vector: np.ndarray
    ) -> float:
        """
        计算viewer对target的单向偏好概率
        
        基于用户声明的偏好 + 实际行为推断的偏好
        """
        gender = viewer_vector[3]
        
        # 获取用户声明的偏好范围
        age_pref_min = viewer_vector[1]  # 年龄偏好下限
        age_pref_max = viewer_vector[2]  # 年龄偏好上限
        
        # 目标的实际值
        target_age = target_vector[0]
        target_income = target_vector[12]
        target_height = target_vector[11]
        
        preference_score = 1.0
        
        # 检查是否符合年龄偏好
        if target_age < age_pref_min or target_age > age_pref_max:
            preference_score *= 0.5  # 不符合年龄偏好，概率降低
        
        # 检查收入偏好（如果有）
        income_pref = viewer_vector.get("income_preference", 0.3)
        if target_income < income_pref:
            preference_score *= 0.7
        
        # 检查身高偏好（女性视角）
        if gender == 1:  # 女性用户
            height_pref = viewer_vector.get("height_preference", 0.65)
            if target_height < height_pref:
                preference_score *= 0.6
        
        return preference_score
```

```python
class MutualMatchRecommendStrategy:
    """双向匹配推荐策略"""
    
    def recommend(
        self,
        user_id: str,
        candidates: List[str],
        market_service: MarketDynamicsService
    ) -> List[Tuple[str, float, str]]:
        """
        推荐策略：优先推"双向匹配概率高"的用户
        
        Returns:
            [(候选ID, 双向匹配概率, 推荐理由)]
        """
        user_vector = self._get_vector(user_id)
        user_score = market_service.compute_competition_score(user_vector, ...)
        
        results = []
        for candidate_id in candidates:
            candidate_vector = self._get_vector(candidate_id)
            candidate_score = market_service.compute_competition_score(candidate_vector, ...)
            
            # 计算双向匹配概率
            mutual_prob = market_service.compute_mutual_match_probability(
                user_vector, candidate_vector,
                user_score, candidate_score
            )
            
            # 只推荐双向概率 > 阈值的
            if mutual_prob > 0.3:  # 阈值可调
                reason = self._generate_reason(mutual_prob, user_score, candidate_score)
                results.append((candidate_id, mutual_prob, reason))
        
        # 按双向概率排序
        results.sort(key=lambda x: x[1], reverse=True)
        
        return results
    
    def _generate_reason(
        self,
        mutual_prob: float,
        user_score: float,
        candidate_score: float
    ) -> str:
        """生成推荐理由（不暴露竞争分）"""
```

### 4.4 吸引力价值平衡算法（Elo Rating）

> **设计背景**：借鉴游戏排名系统（如国际象棋Elo、游戏天梯），为每个用户计算动态吸引力分数。这**不是歧视**，而是为了保护用户体验，减少低成功率的无效匹配，将有限的曝光资源分配给更有可能产生双向奔赴的组合。

```python
class EloAttractivenessRating:
    """
    吸引力Elo评分系统
    
    原理：
    - 每个用户有一个初始Elo分（如1200）
    - 每次被like → Elo分上升
    - 每次被pass → Elo分下降
    - 分数变化幅度取决于对方的Elo分（被高分用户like，加分更多）
    """
    
    def __init__(self, initial_rating: float = 1200, k_factor: float = 32):
        self.initial_rating = initial_rating
        self.k_factor = k_factor  # K因子：分数变化幅度
    
    def update_rating(
        self,
        user_rating: float,
        opponent_rating: float,
        outcome: float  # 1=被like, 0=被pass, 0.5=无操作
    ) -> float:
        """
        更新用户Elo分
        
        Args:
            user_rating: 用户当前Elo分
            opponent_rating: 操作者（like/pass的人）的Elo分
            outcome: 结果（1=被like, 0=被pass）
            
        Returns:
            新的Elo分
        """
        # 预期得分（基于Elo公式）
        expected_score = 1 / (1 + 10 ** ((opponent_rating - user_rating) / 400))
        
        # 实际得分
        actual_score = outcome
        
        # Elo分变化
        new_rating = user_rating + self.k_factor * (actual_score - expected_score)
        
        return new_rating
    
    def get_attractiveness_tier(self, rating: float) -> str:
        """
        获取吸引力层级（用于分层匹配策略）
        
        注意：这个层级永不展示给用户
        """
        if rating >= 1600:
            return "top_5"      # 前5%，高吸引力
        elif rating >= 1400:
            return "top_20"     # 前20%
        elif rating >= 1200:
            return "average"    # 中等
        elif rating >= 1000:
            return "below_avg"  # 中下
        else:
            return "low"        # 低吸引力
```

**匹配分层策略**：

```python
class MatchLayeringStrategy:
    """
    匹配分层策略：基于吸引力落差调整推荐权重
    
    核心原则：
    - 避免"吸引力落差过大"的强力推荐
    - 不是禁止跨层匹配，而是降低推荐优先级
    - 保护用户体验，减少"被无视"的挫败感
    """
    
    def compute_layer_gap_penalty(
        self,
        user_tier: str,
        candidate_tier: str
    ) -> float:
        """
        计算层级差距惩罚系数
        
        Returns:
            惩罚系数（0-1，越小惩罚越重）
        """
        tier_order = ["top_5", "top_20", "average", "below_avg", "low"]
        
        user_idx = tier_order.index(user_tier)
        candidate_idx = tier_order.index(candidate_tier)
        
        gap = abs(user_idx - candidate_idx)
        
        # 层级差距惩罚
        if gap == 0:
            return 1.0   # 同层，无惩罚
        elif gap == 1:
            return 0.85  # 相邻层，轻微惩罚
        elif gap == 2:
            return 0.6   # 跨2层，中等惩罚
        elif gap == 3:
            return 0.35  # 跨3层，重惩罚
        else:
            return 0.15  # 跨4层以上，极重惩罚
    
    def adjust_recommendation_by_layer(
        self,
        base_score: float,
        user_tier: str,
        candidate_tier: str,
        compatibility_score: float  # 匹配度分数
    ) -> float:
        """
        调整推荐分数：考虑层级差距 + 匹配度
        
        核心逻辑：
        - 层级差距大 → 降低推荐分
        - 但匹配度高 → 可以部分弥补
        """
        layer_penalty = self.compute_layer_gap_penalty(user_tier, candidate_tier)
        
        # 匹配度可以弥补层级差距（最多弥补50%的惩罚）
        compatibility_bonus = min(compatibility_score * 0.5, 0.5)
        effective_penalty = layer_penalty + (1 - layer_penalty) * compatibility_bonus
        
        return base_score * effective_penalty
```

```python
        """

        # 检查价值观契合度

            user_vector[16:32], candidate_vector[16:32]
        )
        if values_similarity > 0.8:

            user_vector[64:72], candidate_vector[64:72]

        if growth_similarity > 0.7:
```

### 4.6 隐性偏好与社会期许偏差的博弈

> **设计背景**：很多女生嘴上说喜欢"高富帅"，但实际互动数据显示，她与"性格温和、情绪价值高"的男生聊天更久、更愉快。系统通过**渐进式权重调整**，实现"比你更懂你"。

```python
class ImplicitPreferenceEvolution:
    """
    隐性偏好渐进式权重调整
    
    策略：
    - 初次匹配：参考用户"声明偏好"（权重高）
    - 后期复盘：逐渐提升"行为推断向量"权重
    - 最终目标：推真正能让用户幸福的对象，而非满足虚荣心的对象
    """
    
    def compute_effective_preference_weight(
        self,
        user_id: str,
        interaction_count: int,  # 互动次数
        behavior_data: dict
    ) -> dict:
        """
        计算有效偏好权重
        
        Returns:
            {
                "声明偏好权重": 0.X,
                "行为推断权重": 0.X,
                "隐性特征权重": 0.X
            }
        """
        # 基于互动次数动态调整权重
        if interaction_count < 5:
            # 初期：声明偏好主导
            return {
                "声明偏好权重": 0.7,
                "行为推断权重": 0.2,
                "隐性特征权重": 0.1
            }
        elif interaction_count < 20:
            # 中期：逐渐平衡
            return {
                "声明偏好权重": 0.5,
                "行为推断权重": 0.35,
                "隐性特征权重": 0.15
            }
        else:
            # 后期：行为推断主导
            return {
                "声明偏好权重": 0.3,
                "行为推断权重": 0.5,
                "隐性特征权重": 0.2
            }
    
    def analyze_preference_behavior_gap(
        self,
        user_id: str,
        stated_preference: dict,
        actual_behavior: dict
    ) -> dict:
        """
        分析"声明偏好"与"实际行为"的差异
        
        例如：
        声明：喜欢高收入（>30k）
        实际：与收入15k但情绪价值高的男生聊天时长是30k的3倍
        
        返回：
        {
            "gap_type": "income_emotional_value",
            "insight": "你可能更看重情绪价值而非收入",
            "recommendation": "建议关注情绪价值高的用户"
        }
        """
        gaps = []
        
        # 检查收入偏好 vs 实际行为
        if stated_preference.get("min_income", 0) > 0.5:
            # 用户声明喜欢高收入
            high_income_chat_time = actual_behavior.get("high_income_chat_time", 0)
            low_income_chat_time = actual_behavior.get("low_income_chat_time", 0)
            
            if low_income_chat_time > high_income_chat_time * 2:
                gaps.append({
                    "dimension": "income",
                    "gap_type": "income_emotional_value",
                    "insight": "你实际与情绪价值高的用户互动更多",
                    "recommendation": "可能情绪价值对你更重要"
                })
        
        return gaps
```

### 4.7 跨阶层匹配的幸福感分析

> **设计背景**：研究发现，跨阶层的"高攀型"匹配往往伴随着更低的关系满意度。系统向用户推送"幸福感报告"，展示与自己向量更匹配的用户在一起后的长期幸福感更高。

```python
class CrossTierHappinessAnalysis:
    """
    跨阶层匹配幸福感分析
    
    研究发现：
    - "高攀型"匹配（低分用户匹配高分用户）满意度更低
    - "匹配型"（同层级或相近层级）长期幸福感更高
    """
    
    def compute_happiness_prediction(
        self,
        user_score: float,
        candidate_score: float,
        compatibility_score: float
    ) -> dict:
        """
        计算幸福感预测
        
        Returns:
            {
                "predicted_satisfaction": 0.X,  # 预测满意度
                "happiness_level": "high/medium/low",
                "analysis": "分析说明"
            }
        """
        score_gap = abs(user_score - candidate_score)
        
        # 层级差距对满意度的影响
        if score_gap < 10:
            gap_factor = 1.0   # 同层级，无影响
            analysis = "你们条件相当，更容易建立平等关系"
        elif score_gap < 20:
            gap_factor = 0.9
            analysis = "条件有一定差距，但可以通过沟通弥补"
        elif score_gap < 30:
            gap_factor = 0.75
            analysis = "条件差距较大，可能存在权力不平衡风险"
        else:
            gap_factor = 0.6
            analysis = "条件差距很大，长期维持关系需要更多努力"
        
        # 匹配度可以弥补层级差距
        compatibility_bonus = compatibility_score * 0.3
        
        predicted_satisfaction = (
            gap_factor * 0.6 + 
            compatibility_score * 0.3 +
            compatibility_bonus * 0.1
        )
        
        if predicted_satisfaction > 0.75:
            happiness_level = "high"
        elif predicted_satisfaction > 0.5:
            happiness_level = "medium"
        else:
            happiness_level = "low"
        
        return {
            "predicted_satisfaction": predicted_satisfaction,
            "happiness_level": happiness_level,
            "analysis": analysis
        }
    
    def generate_happiness_report(
        self,
        user_id: str,
        recommendations: List[dict]
    ) -> str:
        """
        生成幸福感报告
        
        向用户展示：与自己匹配度高的用户，长期幸福感更高
        而非单纯追求高分用户
        """
        # 分析用户的推荐列表
        high_score_low_happiness = []  # 高分低幸福感
        medium_score_high_happiness = []  # 中等分高幸福感
        
        for rec in recommendations:
            happiness_pred = self.compute_happiness_prediction(
                rec["user_score"],
                rec["candidate_score"],
                rec["compatibility_score"]
            )
            
            if rec["candidate_score"] > rec["user_score"] + 20:
                if happiness_pred["happiness_level"] == "low":
                    high_score_low_happiness.append(rec)
            elif rec["candidate_score"] < rec["user_score"] + 10:
                if happiness_pred["happiness_level"] == "high":
                    medium_score_high_happiness.append(rec)
        
        report = f"""
        📊 你的匹配幸福感报告
        
        我们分析了你的偏好和匹配数据，发现：
        
        🔍 你可能喜欢的"高分用户"：
        - 虽然外在条件出众
        - 但预测长期满意度可能较低
        - 原因：条件差距可能导致关系不平等
        
        💡 更适合你的"潜力股"：
        - 条件与你相当或略优
        - 内在契合度更高
        - 预测长期幸福感更高
        
        💬 建议：
        长期幸福比短期满足更重要。
        那些真正能让你幸福的，可能不是最耀眼的，
        而是最能理解你、支持你的人。
        """
        
        return report
```

### 4.8 资源调控：防止"公地悲剧"

> **设计背景**：如果所有人都去竞争前5%的用户，会导致：
> 1. 高分用户被大量消息淹没，可能流失
> 2. 普通用户持续被忽视，挫败感强
> 3. 整体匹配效率下降
> 
> 系统需要通过**流量调控**，实现资源均衡分配。

```python
class ResourceAllocationController:
    """
    资源调控服务：防止"公地悲剧"
    
    核心机制：
    1. 限流：高分用户每日被推荐次数上限
    2. 降权：72小时无深度互动 → 降低推荐权重
    3. 引流：向用户推荐"潜力股"而非仅高分用户
    """
    
    def __init__(self):
        self.daily_recommend_limit = {
            "top_5": 50,     # 前5%用户，每日最多被推荐50次
            "top_20": 100,   # 前20%，每日最多100次
            "average": 200,  # 中等用户，每日最多200次
        }
        
        self.depth_stagnant_threshold = 72  # 72小时
    
    def check_recommendation_quota(
        self,
        user_id: str,
        user_tier: str
    ) -> bool:
        """
        检查用户今日是否还有推荐配额
        
        Returns:
            True=可以继续推荐，False=已达上限
        """
        daily_count = self._get_daily_recommend_count(user_id)
        limit = self.daily_recommend_limit.get(user_tier, 200)
        
        return daily_count < limit
    
    def adjust_recommendation_by_engagement(
        self,
        user_id: str,
        base_weight: float
    ) -> float:
        """
        根据互动深度调整推荐权重
        
        逻辑：
        - 72小时内聊天深度无提升 → 降权
        - 互动深度持续增长 → 提权
        """
        # 获取用户72小时互动深度
        depth_metrics = self._get_72h_depth_metrics(user_id)
        
        if depth_metrics["status"] == "stagnant":
            # 深度停滞，降权50%
            return base_weight * 0.5
        
        elif depth_metrics["status"] == "declining":
            # 深度下降，降权70%
            return base_weight * 0.3
        
        elif depth_metrics["status"] == "healthy_growth":
            # 健康增长，提权20%
            return base_weight * 1.2
        
        return base_weight
    
    def get_potential_stock_recommendations(
        self,
        user_id: str,
        user_vector: np.ndarray,
        all_users: List[dict]
    ) -> List[dict]:
        """
        获取"潜力股"推荐
        
        定义：
        - 条件中等但匹配度高
        - 有意愿深度交流
        - 互动深度增长率高
        """
        potentials = []
        
        for candidate in all_users:
            # 跳过高分用户（他们已经被过度曝光）
            if candidate["tier"] in ["top_5", "top_20"]:
                continue
            
            # 计算匹配度
            compatibility = self._compute_compatibility(
                user_vector, candidate["vector"]
            )
            
            # 检查互动意愿
            if candidate["engagement_willingness"] > 0.6:
                potentials.append({
                    "user_id": candidate["id"],
                    "compatibility": compatibility,
                    "engagement_willingness": candidate["engagement_willingness"],
                    "reason": "有意愿深度交流，匹配度高"
                })
        
        # 按匹配度和互动意愿排序
        potentials.sort(
            key=lambda x: x["compatibility"] * 0.6 + x["engagement_willingness"] * 0.4,
            reverse=True
        )
        
        return potentials[:10]  # 返回前10个潜力股
    
    def trigger_rule_review_for_stagnant(
        self,
        user_id: str,
        depth_metrics: dict
    ) -> None:
        """
        当高分用户72小时深度停滞时，触发规则复盘
        
        动作：
        1. 减少该用户流量引入
        2. 向推荐该用户的人推送"潜力股"
        3. 记录案例供LLM学习
        """
        if depth_metrics["status"] == "stagnant":
            # 1. 降权
            self._reduce_user_weight(user_id, factor=0.5)
            
            # 2. 获取推荐过该用户的人
            recommenders = self._get_recommenders(user_id)
            
            # 3. 向他们推送潜力股
            for recommender_id in recommenders:
                potentials = self.get_potential_stock_recommendations(
                    recommender_id,
                    self._get_user_vector(recommender_id),
                    self._get_all_users()
                )
                
                # 推送通知
                self._send_potential_notification(recommender_id, potentials)
            
            # 4. 记录案例
            self._record_stagnant_case(user_id, depth_metrics)
```

```python
class UserVectorEncoder:
```

### 5.2 匹配规则定义

> **设计说明**：在婚恋匹配中，有些组合是"剧毒"的，例如"严重的自恋倾向 × 极度低自尊"或"焦虑型 × 回避型"。如果仅仅是降低权重或增加距离，系统仍可能将其匹配。因此，在 MatchRule 中增加 **BlacklistRegion（黑名单区域）**，对于某些特定的向量组合实行一票否决制，而非加权计算。

```python
@dataclass
class MatchRule:
    """匹配规则"""

    trigger_condition: dict         # 触发条件
    region_definition: dict         # 区域定义
    blacklist_regions: List[dict]   # 黑名单区域（一票否决）
    confidence: float               # 规则置信度
    support_count: int              # 支持案例数
    learned_from: List[str]         # 学习来源


@dataclass
class BlacklistRegion:
    """黑名单区域：一票否决的不可调和组合"""

    learned_from: List[str]           # 学习来源案例
```

### 5.4 匹配函数实现

```python
class MatchFunctionService:
    """匹配函数服务"""
    
    def __init__(self):
        self.rules: List[MatchRule] = []
        self.load_rules()
    
    def compute_match_regions(self, user_vector: np.ndarray) -> List[MatchRegion]:
        """
        计算用户的匹配区域
        
        Args:
            user_vector: 用户向量
            
        Returns:
            匹配区域列表
        """
        regions = []
        
        # 遍历所有规则，找到触发的规则
        for rule in self.rules:
            if self._check_trigger(rule.trigger_condition, user_vector):
                # 生成匹配区域
                for region_def in rule.region_definition.values():
                    region = self._build_region(region_def, user_vector)
                    regions.append(region)
        
        # 如果没有规则触发，使用默认区域
        if not regions:
            regions = self._get_default_regions(user_vector)
        
        return regions
    
    def _check_trigger(self, condition: dict, vector: np.ndarray) -> bool:
        """检查是否触发规则"""
        dimension = condition["dimension"]
        operator = condition["operator"]
        value = condition["value"]
        
        dim_value = self._get_dimension_value(dimension, vector)
        
        if operator == ">":
            return dim_value > value
        elif operator == "<":
            return dim_value < value
        elif operator == ">=":
            return dim_value >= value
        elif operator == "<=":
            return dim_value <= value
        elif operator == "==":
            return abs(dim_value - value) < 0.1
        return False
    
    def _build_region(self, region_def: dict, user_vector: np.ndarray) -> MatchRegion:
        """构建匹配区域"""
        # 根据条件计算区域中心点
        center = np.zeros(176)
        
        for condition in region_def.get("conditions", []):
            dimension = condition["dimension"]
            if "range" in condition:
                # 区域中心取范围中点
                value = (condition["range"][0] + condition["range"][1]) / 2
            elif "value" in condition:
                value = condition["value"]
            else:
                continue
            
            # 设置对应维度
            self._set_dimension_value(dimension, value, center)
        
        # 其他维度继承用户向量的值（相似部分）
        for i in range(176):
            if center[i] == 0:
                center[i] = user_vector[i]
        
        return MatchRegion(
            center=center,
            radius=region_def.get("radius", 0.2),
            weight=region_def.get("weight", 0.5),
            region_type=region_def.get("type", "mixed"),
            description=region_def.get("description", ""),
        )
    
    def check_blacklist(self, user_a_vector: np.ndarray, user_b_vector: np.ndarray) -> Tuple[bool, Optional[str]]:
        """
        检查是否命中黑名单区域（一票否决）
        
        Args:
            user_a_vector: 用户A向量
            user_b_vector: 用户B向量
            
        Returns:
            (是否命中黑名单, 原因说明)
        """
        for blacklist in self.blacklist_regions:
            # 检查两个用户的向量是否都满足黑名单条件
            a_matches = self._check_blacklist_conditions(blacklist.dimension_conditions, user_a_vector)
            b_matches = self._check_blacklist_conditions(blacklist.dimension_conditions, user_b_vector)
            
            # 如果两个用户都命中黑名单条件，则一票否决
            if a_matches and b_matches:
                return True, blacklist.reason
            
            # 交叉检查：A命中某个条件，B命中另一个条件
            if self._check_cross_blacklist(blacklist.dimension_conditions, user_a_vector, user_b_vector):
                return True, blacklist.reason
        
        return False, None
    
    def _check_blacklist_conditions(self, conditions: List[dict], vector: np.ndarray) -> bool:
        """检查向量是否满足所有黑名单条件"""
        matches = 0
        for condition in conditions:
            if self._check_trigger(condition, vector):
                matches += 1
        return matches == len(conditions)
    
    def _check_cross_blacklist(self, conditions: List[dict], vec_a: np.ndarray, vec_b: np.ndarray) -> bool:
        """交叉检查黑名单：A命中某条件，B命中另一条件"""
        a_hits = [self._check_trigger(c, vec_a) for c in conditions]
        b_hits = [self._check_trigger(c, vec_b) for c in conditions]
        
        # 至少各命中一个条件，且合计命中所有条件
        return any(a_hits) and any(b_hits) and (sum(a_hits) + sum(b_hits) >= len(conditions))
```

```python
class VectorIndexService:
    """向量索引服务"""
    
    def __init__(self, dimension: int = 176):
        self.dimension = dimension
        self.index = None
        self.user_id_map = {}  # 向量索引 -> 用户ID
    
    def build_index(self, user_vectors: Dict[str, np.ndarray]):
        """构建向量索引"""
        import faiss
        
        # 收集所有向量
        vectors = []
        user_ids = []
        for user_id, vector in user_vectors.items():
            vectors.append(vector)
            user_ids.append(user_id)
        
        vectors = np.array(vectors).astype('float32')
        
        # 创建 FAISS 索引
        self.index = faiss.IndexFlatL2(self.dimension)
        self.index.add(vectors)
        
        # 建立映射
        self.user_id_map = {i: user_ids[i] for i in range(len(user_ids))}
    
    def search_in_regions(
        self,
        regions: List[MatchRegion],
        exclude_user_ids: List[str] = None,
        limit: int = 100
    ) -> List[Tuple[str, float]]:
        """
        在匹配区域内搜索用户
        
        Args:
            regions: 匹配区域列表
            exclude_user_ids: 排除的用户ID
            limit: 返回数量上限
            
        Returns:
            [(用户ID, 距离), ...]
        """
        results = []
        exclude_set = set(exclude_user_ids or [])
        
        for region in regions:
            # 区域查询：距离中心点不超过半径
            distances, indices = self.index.range_search(
                region.center.reshape(1, -1).astype('float32'),
                region.radius ** 2  # FAISS 使用距离平方
            )
            
            for idx, dist in zip(indices, distances):
                user_id = self.user_id_map.get(idx)
                if user_id and user_id not in exclude_set:
                    # 加权距离
                    weighted_dist = dist * (1 - region.weight)
                    results.append((user_id, weighted_dist))
        
        # 去重，保留最小距离
        unique_results = {}
        for user_id, dist in results:
            if user_id not in unique_results or unique_results[user_id] > dist:
                unique_results[user_id] = dist
        
        # 按距离排序
        sorted_results = sorted(unique_results.items(), key=lambda x: x[1])
        
        return sorted_results[:limit]
```

```python
class MatchRecommendService:
    """匹配推荐服务"""
    
    def __init__(self):
        self.vector_encoder = UserVectorEncoder()
        self.match_function = MatchFunctionService()
        self.vector_index = VectorIndexService()
        self.llm_service = None  # LLM 服务
    
    async def get_recommendations(
        self,
        user_id: str,
        limit: int = 10,
        use_llm_analysis: bool = True
    ) -> List[Recommendation]:
        """
        获取匹配推荐
        
        Args:
            user_id: 用户ID
            limit: 推荐数量
            use_llm_analysis: 是否使用 LLM 深度分析
            
        Returns:
            推荐列表
        """

            raise ValueError(f"User vector not found: {user_id}")

            candidates=candidates
        )
        
        # 5. LLM 深度分析（可选）
        if use_llm_analysis and len(filtered_candidates) > limit:
            final_candidates = await self._llm_analyze(
                user_id=user_id,
                user_vector=user_vector,
                candidates=filtered_candidates[:limit * 2]
            )
        else:
            final_candidates = filtered_candidates[:limit]
        
        # 6. 生成推荐理由
        recommendations = await self._generate_recommendations(
            user_id=user_id,
            candidates=final_candidates,
            match_regions=match_regions
        )
        
        return recommendations[:limit]
    
    async def _get_user_vector(self, user_id: str) -> np.ndarray:
        """获取用户向量"""
        # 从缓存或数据库获取
        pass
    
    async def _apply_hard_filters(
        self,
        user_id: str,
        candidates: List[Tuple[str, float]]
    ) -> List[Tuple[str, float]]:
        """应用硬条件过滤"""
        # 获取用户的硬条件偏好
        user_preferences = await self._get_user_preferences(user_id)
        
        filtered = []
        for candidate_id, distance in candidates:
            candidate = await self._get_user_profile(candidate_id)
            
            # 年龄过滤
            if not self._check_age_range(candidate, user_preferences):
                continue
            
            # 地点过滤
            if not self._check_location(candidate, user_preferences):
                continue
            
            # 关系目标过滤
            if not self._check_relationship_goal(candidate, user_preferences):
                continue
            
            filtered.append((candidate_id, distance))
        
        return filtered
    
    async def _llm_analyze(
        self,
        user_id: str,
        user_vector: np.ndarray,
        candidates: List[Tuple[str, float]]
    ) -> List[Tuple[str, float]]:
        """LLM 深度分析"""
        user_profile = await self._get_user_profile(user_id)
        
        analyzed = []
        for candidate_id, distance in candidates:
            candidate_profile = await self._get_user_profile(candidate_id)
            
            # LLM 分析
            analysis = await self._call_llm_match_analysis(
                user_profile, candidate_profile
            )
            
            if analysis.get("recommend", True):
                # 调整距离（基于 LLM 评分）
                llm_score = analysis.get("score", 0.5)
                adjusted_distance = distance * (2 - llm_score)  # 分数越高，距离越近
                analyzed.append((candidate_id, adjusted_distance, analysis))
        
        # 按调整后距离排序
        analyzed.sort(key=lambda x: x[1])
        
        return [(c[0], c[1]) for c in analyzed]
    
    async def _generate_recommendations(
        self,
        user_id: str,
        candidates: List[Tuple[str, float]],
        match_regions: List[MatchRegion]
    ) -> List[Recommendation]:
        """生成推荐结果"""
        recommendations = []
        
        for candidate_id, distance in candidates:
            candidate_profile = await self._get_user_profile(candidate_id)
            
            # 确定匹配区域类型
            region_type = self._determine_region_type(
                candidate_id, match_regions
            )
            
            # 生成匹配理由
            reasoning = await self._generate_reasoning(
                user_id, candidate_id, region_type
            )
            
            recommendation = Recommendation(
                user_id=candidate_id,
                score=1 - min(distance, 1),  # 距离转分数
                match_type=region_type,
                reasoning=reasoning,
                user_profile=candidate_profile,
            )
            recommendations.append(recommendation)
        
        return recommendations
```

### 12.2 互动深度增长率计算

```python
class InteractionDepthMonitor:
    """互动深度监控服务"""
    
    async def check_72h_depth_growth(
        self,
        match_record_id: str,
        user_a_id: str,
        user_b_id: str
    ) -> DepthGrowthResult:
        """
        检查匹配后72小时的互动深度增长
        
        匹配不是一次性的"买卖"，而是长期的过程。
        如果72小时内聊天深度没有显著提升，系统应反馈复盘。
        
        Args:
            match_record_id: 匹配记录ID
            user_a_id: 用户A ID
            user_b_id: 用户B ID
            
        Returns:
            深度增长检测结果
        """
        # 获取匹配后72小时内的聊天数据
        messages = await self._get_messages_since_match(
            user_a_id, user_b_id, hours=72
        )
        
        if len(messages) < 10:  # 消息太少，无法判断
            return DepthGrowthResult(
                match_record_id=match_record_id,
                status="insufficient_data",
                action="extend_monitoring"  # 继续监控
            )
        
        # NLP分析聊天深度（基于沟通维度120-135）
        depth_metrics = await self._analyze_conversation_depth(messages)
        
        # 计算深度增长率
        initial_depth = depth_metrics["first_24h_depth"]
        current_depth = depth_metrics["current_depth"]
        growth_rate = (current_depth - initial_depth) / initial_depth
        
        # 判断是否需要复盘
        if growth_rate < 0.2:  # 增长率低于20%
            # 反馈给 CaseCollectionService 进行复盘
            await self._trigger_rule_review(
                match_record_id=match_record_id,
                reason="72h_interaction_depth_stagnant",
                depth_metrics=depth_metrics
            )
            
            return DepthGrowthResult(
                match_record_id=match_record_id,
                status="depth_stagnant",
                growth_rate=growth_rate,
                action="review_match_rule"  # 触发规则复盘
            )
        
        return DepthGrowthResult(
            match_record_id=match_record_id,
            status="healthy_growth",
            growth_rate=growth_rate,
            action="continue_monitoring"
        )
    
    async def _analyze_conversation_depth(self, messages: List[dict]) -> dict:
        """
        NLP分析聊天深度
        
        基于沟通维度 (128-143):
        - v136-v139: 话题偏好（话题多样性）
        - v140-v143: 冲突修复能力、沟通主动性
        """
        # 话题多样性（涉猎的话题种类）
        topic_diversity = self._calculate_topic_diversity(messages)
        
        # 情感深度（是否涉及情感、价值观等深层话题）
        emotional_depth = self._calculate_emotional_depth(messages)
        
        # 互动主动性（双方是否都主动发起话题）
        initiative_balance = self._calculate_initiative_balance(messages)
        
        # 整体深度评分
        overall_depth = (
            topic_diversity * 0.3 +
            emotional_depth * 0.4 +
            initiative_balance * 0.3
        )
        
        return {
            "first_24h_depth": overall_depth,  # 前24小时深度
            "current_depth": overall_depth,
            "topic_diversity": topic_diversity,
            "emotional_depth": emotional_depth,
            "initiative_balance": initiative_balance
        }
```

```python
class MatchEvaluator:

        Args:
            test_cases: 测试案例
            
        Returns:
            评估结果
        """
        correct = 0
        total = len(test_cases)
        
        for case in test_cases:
            # 预测
            regions = self.match_function.compute_match_regions(case.user_a_vector)
            is_match = self._is_in_regions(case.user_b_vector, regions)

            if is_match == (case.outcome == "success"):
                correct += 1
        
        accuracy = correct / total if total > 0 else 0
        
        return EvalResult(
            accuracy=accuracy,
            total_cases=total,
            correct_cases=correct,
        )
```

## 十三、风险分析与改进建议

> **设计说明**：任何算法系统都存在潜在的隐患和盲区。本章节识别向量匹配系统可能存在的风险，并提出改进方案，确保系统既能帮助用户找到匹配对象，又能避免"算法比现实更冷酷"的负面效应。

### 13.1 隐藏的隐患：算法可能比现实更冷酷

#### 13.1.1 "信息茧房"风险（隐性偏好陷阱）

**问题描述**：

系统在互动20次后大幅提升"行为推断"权重（见 §4.6）。如果一个用户近期状态低迷，频繁点击"高颜值但'海王'类型"寻求刺激，系统会误以为这是 TA 的真爱，从而不断推送同类，导致用户在"垃圾关系"里循环，无法实现方案中提到的"成长"。

```
问题路径：
├── 用户近期情绪低谷（v152-159状态向量异常）
├── 行为数据：频繁点击"高颜值+高风险"类型
├── 行为推断权重提升（互动>20次后权重50%）
├── 系统误判：认为这是用户的"真实偏好"
└── 结果：持续推送同类，形成"信息茧房"
```

**改进方案：引入"随机扰动因子"（探索位）**

```python
class ExplorationSlotAllocator:
    """
    探索位分配器：防止信息茧房
    
    核心机制：
    - 每天保留10%的推荐位，完全跳出算法预测
    - 推送一些"算法认为你不喜欢但实际上对你有益"的类型
    - 防止用户审美疲劳，打破恶性循环
    """
    
    def __init__(self):
        self.exploration_ratio = 0.1  # 10%探索位
        self.exploration_candidates = []
    
    def allocate_exploration_slots(
        self,
        user_id: str,
        user_vector: np.ndarray,
        regular_recommendations: List[dict],
        all_candidates: List[dict]
    ) -> List[dict]:
        """
        分配探索位
        
        逻辑：
        1. 从算法未推荐的用户中，选取匹配度高但偏好度低的
        2. 优先选取"成长型用户"（高成长意愿v64-71）
        3. 优先选取"情绪稳定用户"（高安全型得分v48）
        """
        # 已推荐的用户ID
        recommended_ids = {r["user_id"] for r in regular_recommendations}
        
        # 剩余候选
        remaining = [c for c in all_candidates if c["user_id"] not in recommended_ids]
        
        # 探索位候选筛选条件
        exploration_candidates = []
        for candidate in remaining:
            candidate_vector = candidate["vector"]
            
            # 计算匹配度（基于价值观、成长意愿等深层维度）
            deep_compatibility = self._compute_deep_compatibility(
                user_vector, candidate_vector
            )
            
            # 计算偏好偏离度（算法认为你不喜欢）
            preference_deviation = self._compute_preference_deviation(
                user_vector, candidate_vector
            )
            
            # 筛选条件：匹配度高 + 偏离度高 = 探索位候选
            if deep_compatibility > 0.6 and preference_deviation > 0.3:
                # 检查候选是否为"有益型"（高成长意愿、安全型）
                growth_score = candidate_vector[64:72].mean()
                secure_score = candidate_vector[48]
                
                if growth_score > 0.5 or secure_score > 0.6:
                    exploration_candidates.append({
                        "user_id": candidate["user_id"],
                        "deep_compatibility": deep_compatibility,
                        "preference_deviation": preference_deviation,
                        "growth_score": growth_score,
                        "secure_score": secure_score,
                        "reason": "探索推荐：可能比你想象中更适合"
                    })
        
        # 按深层匹配度排序
        exploration_candidates.sort(
            key=lambda x: x["deep_compatibility"] * 0.6 + x["growth_score"] * 0.4,
            reverse=True
        )
        
        # 返回10%的探索位
        exploration_count = int(len(regular_recommendations) * self.exploration_ratio)
        return exploration_candidates[:exploration_count]
    
    def _compute_deep_compatibility(
        self,
        user_vector: np.ndarray,
        candidate_vector: np.ndarray
    ) -> float:
        """
        计算深层匹配度
        
        基于：
        - 价值观维度 [16-31]
        - 成长意愿维度 [64-71]
        - 依恋类型维度 [48-63]
        """
        values_similarity = 1 - np.linalg.norm(
            user_vector[16:32] - candidate_vector[16:32]
        ) / np.sqrt(16)
        
        growth_similarity = 1 - np.linalg.norm(
            user_vector[64:72] - candidate_vector[64:72]
        ) / np.sqrt(8)
        
        # 安全型得分加成
        secure_bonus = min(candidate_vector[48], user_vector[48]) * 0.2
        
        return values_similarity * 0.4 + growth_similarity * 0.4 + secure_bonus + 0.2
    
    def _compute_preference_deviation(
        self,
        user_vector: np.ndarray,
        candidate_vector: np.ndarray
    ) -> float:
        """
        计算偏好偏离度
        
        基于用户声明偏好 vs 候选实际特征
        """
        # 用户偏好：高颜值、高收入（常见"刺激型"偏好）
        user_visual_pref = user_vector.get("visual_preference", 0.5)
        user_income_pref = user_vector.get("income_preference", 0.5)
        
        # 候选特征
        candidate_visual = candidate_vector[168]  # 被喜欢率（间接反映外貌）
        candidate_income = candidate_vector[12]
        
        # 偏离度：候选不符合用户偏好
        visual_deviation = abs(candidate_visual - user_visual_pref)
        income_deviation = abs(candidate_income - user_income_pref)
        
        return visual_deviation * 0.5 + income_deviation * 0.5
```

#### 13.1.2 "市场竞争分"的物化反弹

**问题描述**：

虽然市场竞争分（v168-175）不展示给用户，但算法逻辑（如男性看收入、女性看年龄/外貌）非常传统（见 §4.2.13）。这可能加剧社会偏见，让一些"非典型成功"用户（如全职奶爸、事业型女性）在系统里彻底失去曝光。

```
问题表现：
├── 男性评分：身高×30% + 收入×35% + 房×10% + 车×5% + 学历×10%
├── 女性评分：年龄×35%（年轻=高分） + 外貌×30% + 学历×15%
├── 全职奶爸：收入维度得分低 → 竞争分低 → 曝光减少
├── 事业型女性：年龄维度得分低 → 竞争分低 → 曝光减少
└── 结果：系统加剧传统偏见，"非典型成功者"被边缘化
```

**改进方案：增加"多元审美通道"**

```python
class DiverseAestheticChannel:
    """
    多元审美通道：允许用户自定义权重
    
    核心机制：
    - 用户可自定义匹配权重偏好
    - 例如："我就是不在乎对方收入，更在乎情绪价值"
    - 用户个性化选择能修正系统的"大盘评分"
    """
    
    def __init__(self):
        # 默认权重（传统评分）
        self.default_weights = {
            "male": {"height": 0.30, "income": 0.35, "house": 0.10, "car": 0.05, "education": 0.10},
            "female": {"age": 0.35, "appearance": 0.30, "education": 0.15, "height": 0.05, "income": 0.10}
        }
        
        # 用户自定义权重存储
        self.user_custom_weights = {}
    
    def set_custom_weights(
        self,
        user_id: str,
        custom_weights: dict
    ) -> None:
        """
        设置用户自定义权重
        
        示例：
        {
            "income": 0.05,       # 降低收入权重
            "emotional_value": 0.40,  # 提升情绪价值权重
            "values_match": 0.30,
            "growth_willingness": 0.25
        }
        """
        self.user_custom_weights[user_id] = custom_weights
    
    def compute_personalized_score(
        self,
        user_id: str,
        user_vector: np.ndarray,
        candidate_vector: np.ndarray,
        behavior_data: dict
    ) -> float:
        """
        计算个性化评分
        
        如果用户设置了自定义权重，使用自定义权重
        否则使用默认权重
        """
        weights = self.user_custom_weights.get(user_id, self._get_default_weights(user_vector))
        
        score = 0.0
        
        # 传统维度
        if "height" in weights:
            score += candidate_vector[11] * weights["height"]
        if "income" in weights:
            score += candidate_vector[12] * weights["income"]
        if "education" in weights:
            score += candidate_vector[10] * weights["education"]
        
        # 新增多元维度
        if "emotional_value" in weights:
            # 情绪价值 = 安全型得分 + 宜人性 + 情绪稳定性
            emotional_value = (
                candidate_vector[48] * 0.4 +  # 安全型得分
                candidate_vector[45] * 0.3 +  # 宜人性
                candidate_vector[46] * 0.3    # 情绪稳定性
            )
            score += emotional_value * weights["emotional_value"]
        
        if "values_match" in weights:
            # 价值观匹配度（需要在匹配时计算）
            values_similarity = 1 - np.linalg.norm(
                user_vector[16:32] - candidate_vector[16:32]
            ) / np.sqrt(16)
            score += values_similarity * weights["values_match"]
        
        if "growth_willingness" in weights:
            # 成长意愿
            growth_score = candidate_vector[64:72].mean()
            score += growth_score * weights["growth_willingness"]
        
        return min(score, 1.0)
    
    def suggest_weight_adjustment(
        self,
        user_id: str,
        behavior_data: dict
    ) -> dict:
        """
        根据用户行为数据，建议权重调整
        
        示例：
        如果用户与"情绪价值高"的用户互动更多，
        系统建议：提升情绪价值权重
        """
        # 分析用户互动偏好
        interaction_analysis = self._analyze_interaction_patterns(behavior_data)
        
        suggestions = {}
        
        if interaction_analysis["emotional_value_interaction_ratio"] > 0.6:
            suggestions["emotional_value"] = {
                "suggested_weight": 0.35,
                "reason": "你与情绪价值高的用户互动更多，可能情绪价值对你更重要"
            }
        
        if interaction_analysis["income_interaction_ratio"] < 0.3:
            suggestions["income"] = {
                "suggested_weight": 0.10,
                "reason": "你似乎不太关注对方的收入水平"
            }
        
        return suggestions
```

#### 13.1.3 LLM的"一本正经胡说八道"

**问题描述**：

方案极度依赖 LLM 分析"声明与行为背离"（见 §4.2.10）。如果 LLM 误读了用户的幽默感或讽刺语气，给出的"心理疏导式"推荐理由可能会显得傲慢且令人反感。

```
问题场景：
├── 用户说："我喜欢高富帅"（幽默/自嘲语气）
├── LLM误判：认为用户真实偏好是"高富帅"
├── 隐性特征分析：标记用户为"物质导向"
├── 推荐理由："你可能更看重物质条件" ← 傲慢！
└── 用户反应："我只是开玩笑，系统怎么这么刻板？"
```

**改进方案：引入语气识别与置信度校验**

```python
class LLMIntentAnalyzer:
    """
    LLM意图分析器：防止误读幽默/讽刺
    
    核心机制：
    - 语气识别：区分真实表达 vs 幽默/讽刺
    - 置信度校验：低置信度时标注"不确定"，不强行下结论
    - 用户反馈闭环：记录用户对推荐理由的反应
    """
    
    def __init__(self):
        self.tone_classifier = None  # 语气分类模型
        self.confidence_threshold = 0.7  # 置信度阈值
    
    async def analyze_user_expression(
        self,
        user_id: str,
        expression: str,
        context: dict
    ) -> dict:
        """
        分析用户表达
        
        区分：
        - 真实偏好表达
        - 幽默/自嘲表达
        - 社会期许偏差（说场面话）
        - 不确定（需要更多数据）
        """
        # 语气分类
        tone = await self._classify_tone(expression)
        
        # 语义分析
        semantic = await self._analyze_semantic(expression)
        
        # 上下文校验（行为数据是否支撑）
        behavior_support = await self._check_behavior_support(
            user_id, semantic["inferred_preference"]
        )
        
        # 置信度计算
        confidence = self._compute_confidence(tone, semantic, behavior_support)
        
        return {
            "expression": expression,
            "tone": tone,  # "serious", "humorous", "sarcastic", "uncertain"
            "semantic": semantic,
            "behavior_support": behavior_support,
            "confidence": confidence,
            "final_judgment": self._make_judgment(tone, semantic, behavior_support, confidence)
        }
    
    async def _classify_tone(self, expression: str) -> str:
        """
        语气分类
        
        使用语气分类模型或LLM判断
        """
        # 特征关键词
        humorous_keywords = ["开玩笑", "哈哈", "玩笑", "调侃", "自嘲"]
        sarcastic_keywords = ["怎么可能", "想得美", "做梦", "鬼才信"]
        
        # 简化版分类
        for kw in humorous_keywords:
            if kw in expression:
                return "humorous"
        
        for kw in sarcastic_keywords:
            if kw in expression:
                return "sarcastic"
        
        # 使用LLM精确分类
        llm_result = await self._llm_tone_classify(expression)
        return llm_result
    
    def _make_judgment(
        self,
        tone: str,
        semantic: dict,
        behavior_support: float,
        confidence: float
    ) -> dict:
        """
        做出最终判断
        
        关键：
        - 语气为幽默/讽刺 → 不作为偏好依据
        - 置信度低 → 标注"不确定"，不强行下结论
        """
        if tone in ["humorous", "sarcastic"]:
            return {
                "use_for_matching": False,
                "reason": "用户表达为幽默/讽刺语气，不作为偏好依据",
                "suggestion": "等待更多真实行为数据"
            }
        
        if confidence < self.confidence_threshold:
            return {
                "use_for_matching": False,
                "reason": f"置信度不足（{confidence:.2f}），避免傲慢判断",
                "suggestion": "继续收集行为数据，不做定性"
            }
        
        if behavior_support < 0.5:
            return {
                "use_for_matching": False,
                "reason": "行为数据不支撑语义分析，可能是社会期许偏差",
                "suggestion": "以行为数据为准，而非声明"
            }
        
        return {
            "use_for_matching": True,
            "reason": "置信度足够，行为数据支撑",
            "inferred_preference": semantic["inferred_preference"]
        }
    
    def generate_humility_reasoning(
        self,
        analysis_result: dict
    ) -> str:
        """
        生成谦逊的推荐理由
        
        避免傲慢语气，承认不确定性
        """
        if analysis_result["confidence"] < self.confidence_threshold:
            return (
                "我们还在学习你的真实偏好。"
                "目前的推荐基于你声明的偏好，"
                "随着更多互动，我们会更了解你真正适合什么类型。"
            )
        
        if analysis_result["tone"] in ["humorous", "sarcastic"]:
            return (
                "你的表达很有趣！"
                "我们注意到这可能不是你的严肃偏好。"
                "让我们通过实际互动来发现你真正适合的人。"
            )
        
        # 高置信度时，仍然保持谦逊
        return (
            f"基于你的互动数据，我们推测{analysis_result['semantic']['inferred_preference']}。"
            "如果这个判断有偏差，请告诉我们，我们会调整推荐策略。"
        )
```

### 13.2 缺失的维度：需要补充的"补丁"

#### 13.2.1 现实社交圈的"排雷"机制

**问题背景**：

婚恋中最稳固的关系往往来自"熟人背书"。如果两人有共同好友，匹配权重应大幅提升；如果有共同的前任或纠纷记录，应进入黑名单。

**改进方案：增加"二度人脉关联"（基于隐私授权）**

```python
class SocialCircleAnalyzer:
    """
    社交圈分析器：二度人脉关联
    
    核心机制：
    - 共同好友 → 匹配权重提升
    - 共同前任/纠纷 → 黑名单标记
    - 基于用户隐私授权
    """
    
    def __init__(self):
        self.friend_bonus_factor = 1.3  # 共同好友权重加成30%
        self.blacklist_patterns = ["shared_ex", "dispute_record", "complaint_history"]
    
    async def analyze_social_connection(
        self,
        user_a_id: str,
        user_b_id: str,
        privacy_settings: dict
    ) -> dict:
        """
        分析社交关联
        
        需要用户授权才能查看社交圈数据
        """
        # 检查隐私授权
        if not privacy_settings.get("allow_social_circle_matching", False):
            return {
                "status": "privacy_restricted",
                "message": "用户未授权社交圈匹配"
            }
        
        # 获取社交圈数据
        a_friends = await self._get_user_friends(user_a_id)
        b_friends = await self._get_user_friends(user_b_id)
        
        # 计算共同好友
        mutual_friends = set(a_friends) & set(b_friends)
        
        # 检查危险关联
        danger_connections = await self._check_danger_connections(user_a_id, user_b_id)
        
        # 计算社交关联权重
        if danger_connections:
            return {
                "status": "blacklist",
                "reason": danger_connections["reason"],
                "action": "exclude_from_matching"
            }
        
        if mutual_friends:
            friend_count = len(mutual_friends)
            bonus = min(self.friend_bonus_factor + friend_count * 0.05, 1.5)  # 最高50%
            
            return {
                "status": "friend_backed",
                "mutual_friends": list(mutual_friends),
```

#### 13.2.3 "反杀猪盘"与诚信分

**问题背景**：

如果一个用户的向量在短时间内剧烈波动（比如突然换了高端人设、疯狂点击高净值人群），系统应识别为"疑似诈骗"或"账号买卖"。

**改进方案：增加"向量偏移预警"**

```python
class VectorDriftMonitor:
    """
    向量偏移监控器：反杀猪盘
    
    核心机制：
    - 监控向量在短时间内的变化幅度
    - 异常波动 → 疑似诈骗/账号买卖
    - 触发人工介入或安全审查
    """
    
    def __init__(self):
        self.drift_threshold = 0.3  # 7天内向量偏移阈值
        self.alert_threshold = 0.5  # 高风险偏移阈值
        self.monitoring_window_days = 7
    
    async def monitor_vector_drift(
        self,
        user_id: str,
        current_vector: np.ndarray,
        historical_vectors: List[np.ndarray]
    ) -> dict:
        """
        监控向量偏移
        
        检查用户向量在最近7天的变化幅度
        """
        if not historical_vectors:
            return {"status": "no_history", "action": "wait_for_data"}
        
        # 获取7天前的向量
        reference_vector = historical_vectors[-1]  # 最新历史向量
        
        # 计算向量偏移幅度
        drift = np.linalg.norm(current_vector - reference_vector) / np.sqrt(len(current_vector))
        
        # 计算关键维度偏移（身份相关维度）
        identity_drift = self._compute_identity_drift(current_vector, reference_vector)
        
        # 计算行为维度偏移（点击偏好）
        behavior_drift = self._compute_behavior_drift(current_vector, reference_vector)
        
        # 判断风险等级
        if drift > self.alert_threshold or identity_drift > 0.4:
            return {
                "status": "high_risk",
                "drift_score": drift,
                "identity_drift": identity_drift,
                "behavior_drift": behavior_drift,
                "action": "security_review",  # 触发安全审查
                "reason": "向量剧烈波动，疑似账号买卖或诈骗"
            }
        
        if drift > self.drift_threshold:
            return {
                "status": "moderate_risk",
                "drift_score": drift,
                "action": "monitor_closely",  # 密切监控
                "reason": "向量变化较大，需持续观察"
            }
        
        return {
            "status": "normal",
            "drift_score": drift,
            "action": "continue_monitoring"
        }
    
    def _compute_identity_drift(
        self,
        current: np.ndarray,
        historical: np.ndarray
    ) -> float:
        """
        计算身份维度偏移
        
        包括：收入、学历、职业等关键身份信息
        """
        identity_dims = [10, 12, 13, 14, 15]  # 学历、收入、职业、房、车
        
        drifts = []
        for dim in identity_dims:
            drift = abs(current[dim] - historical[dim])
            drifts.append(drift)
        
        return max(drifts)  # 取最大偏移
    
    def _compute_behavior_drift(
        self,
        current: np.ndarray,
        historical: np.ndarray
    ) -> float:
        """
        计算行为维度偏移
        
        包括：点击偏好、互动模式
        """
        behavior_dims = range(112, 128)  # 行为模式维度
        
        behavior_drift = np.linalg.norm(
            current[112:128] - historical[112:128]
        ) / np.sqrt(16)
        
        return behavior_drift
    
    async def trigger_security_review(
        self,
        user_id: str,
        drift_analysis: dict
    ) -> None:
        """
        触发安全审查
        
        动作：
        1. 暂停用户匹配权限
        2. 要求重新身份验证
        3. 通知安全团队人工介入
        """
        # 暂停匹配权限
        await self._suspend_matching(user_id)
        
        # 要求重新身份验证
        await self._require_re_verification(user_id)
        
        # 通知安全团队
        await self._notify_security_team(user_id, drift_analysis)
```
