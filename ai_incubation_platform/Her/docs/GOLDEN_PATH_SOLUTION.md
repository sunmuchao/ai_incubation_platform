# 第一条：主路径收敛 — 完整解决方案

> 对应改进项 **IMP-001**：先讲好「一条路」，再挂次要能力，避免功能大杂烩稀释用户心智与研发资源。  
> 与代码现状对齐：`frontend/src/pages/HomePage.tsx`（对话优先 + 抽屉/全屏子功能）、`frontend/src/components/FeaturesDrawer.tsx`（功能列表）、`frontend/src/api/index.ts`（匹配应走 `/api/her/chat`）。

---

## 1. 问题定义（为什么要做）

### 1.1 用户侧

- 入口一多，用户不知道「下一步该干什么」；
- 匹配、滑动、Who Likes Me、认证、会员并列出现时，**核心价值（对话里找人、开聊、被陪跑）被稀释**；
- 新用户认知成本高，留存与首聊转化变差。

### 1.2 产品 / 研发侧

- 每条支线都要维护状态、埋点、异常与文案，**边际成本指数上升**；
- 前后端对「主路径该调哪个 API」若不一致，易出现重复逻辑与 bug（例如对话匹配与列表匹配分叉）。

### 1.3 与当前实现的关系（事实锚点）

- 首页已是 **Chat-first**：`App.tsx` 登录后进 `HomePage`，主视图为 `ChatInterface`；
- 次要能力通过 **`FeaturesDrawer` 的 FEATURES 列表** 与 **`renderView()` 全屏切换**（滑动匹配、Who Likes Me、置信度、人脸等）进入；
- 风险点：抽屉里功能项较多，**与主对话的叙事未显式绑定**（用户可能当「应用商店」用，而不是「红娘流程」）。

---

## 2. 目标状态（验收口径）

### 2.1 北极星主路径（唯一对外叙事）

用一句话对外：**「打开 Her，说话；想见人就说要什么，Her 给你人并陪你聊下去。」**

拆成可观测的 5 步（漏斗）：

| 步骤 | 用户行为 | 系统应做到 |
|------|----------|------------|
| A | 进入首页 | 默认落在**主对话**，不先弹一堆功能 |
| B | 用自然语言表达诉求 | **优先**走 Her 编排：`/api/her/chat`（与 `index.ts` 注释一致） |
| C | 看到推荐/卡片 | 能**一键开聊**（进入 `ChatRoom` 或等价路径） |
| D | 在私聊里 | 悬浮球 / 快捷话术承接「分析对象、约见面、话题」等（与 `CHAT_SCENE_QUICK_OPTIONS` 一致方向） |
| E | 需要高级能力 | 从**对话内引导**或**单一「更多」入口**进入，且能**一键回主对话** |

### 2.2 成功指标（建议最小集）

- **主路径完成率**：进入首页后 24h 内完成「B→C」的比例；
- **首聊率**：完成推荐后 24h 内发起首条私聊的比例；
- **功能抽屉点击率 / 主对话发送量比**：过高说明主路径没讲清；
- **从子页返回主对话的流失**：全屏子流程返回后是否继续发消息。

## 3. 完整方案（三层：叙事 / 交互 / 工程）

### 3.1 产品叙事层（定规矩）

1. **唯一主故事线**  
   对外文档、应用内首屏文案、运营话术，只强调上述 5 步，其它功能一律称为「进阶能力」或「需要时再开」。

2. **功能分级（建议三档）**  
   - **P0 主路径**：对话匹配、会话列表、私聊、Her 悬浮球（与主对话强相关）；  
   - **P1 增长 / 信任**：人脸认证、置信度、Who Likes Me（在「有对象可聊」或「要提升信任」时出现）；  
   - **P2 增值 / 探索**：礼物、里程碑、视频片段、会员等（对话或里程碑节点再引导）。

3. **触发原则**  
   - **默认不打断**：非 P0 不用系统弹窗抢首屏；  
   - **上下文触发**：例如匹配卡片出现后，再轻提示「完成认证更易被推荐」（而非一进 app 就催认证）。

### 3.2 交互与信息架构层（改什么）

1. **首页状态机（概念）**  
   与现有 `renderView()` 一致思想，但对外可文档化为：

   ```text
   MainChat（默认） ⇄ ChatRoom（与某人聊）  
        ↓ 轻入口  
   Drawer / Modal（不替换主栈）  
        ↓ 重任务  
   FullScreen（滑动匹配等）且顶部固定「返回对话」
   ```

2. **`FeaturesDrawer` 收敛策略（两选一或组合）**  
   - **方案 A（推荐）**：抽屉内分两组 ——「常用：与聊天相关」「更多：账户与增值」，默认只展开第一组；  
   - **方案 B**：首访 N 天内只显示 P0+P1，P2 折叠在「更多」里（需本地 `guideStorage` 或后端灰度配合）；  
   - **方案 C**：把若干项从抽屉迁到「个人资料抽屉」里（资料、认证、偏好），主抽屉只留「匹配方式变体」（如滑动）和会员入口。

3. **首访引导（已有 `FeatureGuideModal`）**  
   引导文案只讲主路径 3 步：说话 → 要人 → 开聊；**不要**在首访罗列抽屉里全部功能名。

4. **从子流程返回**  
   全屏页（`SwipeMatchPage` 等）保持显眼「返回」且语义为**回到和 Her 的对话**，而不是「回到空白页」。

### 3.3 工程与契约层（怎么落地、防回归）

1. **API 单一真相（主路径）**  
   - 对话式匹配：**`POST /api/her/chat`**（`ConversationMatchService` 编排）；  
   - 列表/滑动等：**在 PR 与代码评审中规定**「新功能不得绕过 Her 编排私自拼推荐逻辑」；  
   - 文档：在 `frontend/src/api/index.ts` 已有注释处，链接回本文件的「主路径契约」小节（实施时加一行注释即可）。

2. **推荐数据与首屏**  
   - 当前 `getAiPushRecommendations()` 存在「异步触发、立即空返回」与 `HomePage` 期望 `matches.matches` 的潜在不一致，**应纳入主路径专项**（与 IMP-003 联动）：要么统一返回结构并真正填充缓存，要么 UI 明确「加载中/暂无」。

3. **埋点（最小实现）**  
   - 在 `handleFeatureSelect`、`renderView` 分支切换、`handleStartChat`、首条 `chatApi.sendMessage` 打 **同一套 event 名**（如 `golden_path_step_B` …），便于漏斗报表。

4. **灰度（可选）**  
   若已有 `FeatureFlag` / 灰度 API：用开关控制「抽屉默认展开组」「P2 是否可见」，便于 A/B。

## 4. 分阶段实施计划（可直接排期）

### 阶段 0：对齐（0.5～1 天）

- 产品 / 研发 / 设计对表：确认 P0/P1/P2 列表与「主路径 5 步」文案；  
- 在 README 或内部 wiki 贴链接到本文档。

### 阶段 1：低成本收敛（约 3～5 天）

- 改 `FeaturesDrawer`：分组 + 默认折叠「更多」；  
- 改 `FeatureGuideModal` 文案：只讲主路径；  
- 加 4～6 个关键埋点（见 3.3）。

### 阶段 2：数据与体验（约 1～2 周）

- 修复或统一「AI 推送推荐」与首页缓存逻辑（与 IMP-003）；  
- 根据漏斗数据调整：抽屉项顺序、是否首周隐藏 P2。

### 阶段 3：结构性优化（按需）

- 个人中心式抽屉拆分「资料 / 认证 / 偏好」；  
- 子流程更多改为对话内 Generative UI 卡片，减少全屏页数量。

---

## 5. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 收敛过猛，老用户找不到功能 | P2 折叠而非删除；搜索或对话触发「打开匹配偏好」等 |
| 研发仍绕开 `/api/her/chat` | PR checklist + 接口层 Lint/文档链接 |
| 指标缺失无法验证 | 阶段 1 必须上最小埋点 |
