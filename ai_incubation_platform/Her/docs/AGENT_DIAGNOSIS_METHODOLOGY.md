# Agent 系统"智障"诊断方法论

> **适用场景**：LLM + Agent 框架组合表现不如预期，出现理解偏差、工具调用失败、上下文混乱等问题

---

## 一、问题现象分类

| 现象 | 典型表现 | 用户感知 |
|------|---------|---------|
| **理解偏差** | 用户说 A，Agent 理解成 B；过度解读简单指令 | "我说的是 X，你怎么理解成 Y 了？" |
| **工具调用失败** | 该调时不调、不该调时乱调、参数传错 | "为什么你没调用这个工具？"、"你怎么调错了？" |
| **上下文混乱** | 多轮对话后丢失关键信息；被无关信息干扰 | "我刚才明明说了..."、"你怎么忘了？" |
| **输出质量不稳定** | 有时很聪明，有时像换了个脑子 | "刚才还好好的，怎么突然变蠢了？" |
| **无法自我纠错** | 犯错后无法根据反馈调整，重复同样错误 | "我说你错了，你怎么还在犯同样的错？" |
| **被动响应** | 不主动思考，只会机械执行指令 | "你能不能自己想想？"、"你就是个执行器" |

## 二、五问法根因分析

### 分析框架

```
问题现象：[描述具体表现]
├─ 为什么 1: [第一层原因 - 表层]
├─ 为什么 2: [第二层原因 - 机制层]
├─ 为什么 3: [第三层原因 - 设计层]
├─ 为什么 4: [第四层原因 - 架构层]
└─ 为什么 5: [第五层原因 - 根因]

根本对策：[针对根因的解决方案]
```

### 核心维度

| 维度 | 检查点 | 典型根因 |
|------|-------|---------|
| **模型能力** | 模型推理能力、指令遵循能力、上下文理解能力 | 模型本身能力不足 |
| **System Prompt** | 角色定义、决策规则、输出约束、规则冲突 | 规则冲突、层级混乱、内容过多 |
| **工具定义** | 描述清晰度、触发条件、参数设计、边界区分 | 描述模糊、无触发条件、工具边界不清 |
| **上下文管理** | 信息过载、关键信息淹没、记忆机制 | 缺少单一真相来源、信息丢失 |
| **架构设计** | Agent 循环、错误恢复、自我反思机制 | 多层冗余、缺少纠错机制 |

## 三、诊断检查清单

### 3.1 System Prompt 诊断

| 检查项 | 判断标准 | 问题信号 |
|--------|---------|---------|
| **层级一致性** | 多层 Prompt 是否存在规则冲突？ | Agent 不知道该遵循哪层规则 |
| **内容精简度** | 是否超过模型能记住的容量？ | Agent 忽略后半部分规则 |
| **优先级明确性** | 规则优先级是否明确标注？ | Agent 在矛盾规则间摇摆 |
| **覆盖机制** | 业务规则是否明确覆盖框架规则？ | Agent 执行了错误的规则 |

**诊断方法**：
1. 画出 Prompt 层级图（L1 框架 → L2 业务 → L3 Skill）
2. 找出相邻层级的规则冲突点
3. 检查是否有明确的"覆盖"声明

### 3.2 工具定义诊断

| 检查项 | 判断标准 | 问题信号 |
|--------|---------|---------|
| **触发条件** | 是否明确写"何时调用"？ | Agent 不知道该调哪个工具 |
| **禁止场景** | 是否明确写"何时**不**调用"？ | Agent 调错了相似工具 |
| **参数说明** | 参数是否足够清晰？ | Agent 传错参数 |
| **边界区分** | 相似工具是否有边界说明？ | Agent 在相似工具间选择错误 |

**诊断方法**：
1. 列出所有相似功能的工具
2. 检查每个工具是否有 `【触发条件】` 和 `【禁止场景】`
3. 找出容易混淆的工具组合（如 `her_get_icebreaker` vs `her_initiate_chat`）

### 3.3 架构设计诊断

| 检查项 | 判断标准 | 问题信号 |
|--------|---------|---------|
| **冗余层级** | 是否有不必要的中间层？ | 多层路由导致信息丢失 |
| **单一真相来源** | 规则/配置是否有唯一来源？ | 多处配置不一致 |
| **错误恢复** | 是否有用户纠错处理机制？ | Agent 重复错误 |
| **日志追踪** | 关键路径是否有结构化日志？ | 问题难以定位 |

**诊断方法**：
1. 画出数据流图：用户输入 → 各层处理 → 输出
2. 找出超过 3 层的处理链路（简化机会）
3. 找出同一信息的多处定义（统一机会）

## 四、修复策略矩阵

| 根因类型 | 修复策略 | 典型操作 |
|---------|---------|---------|
| **Prompt 层级冲突** | 统一层级，明确覆盖 | SOUL.md 开头写"⚠️ 覆盖框架规则" |
| **Prompt 内容过多** | 精简到核心规则 | 从 500 行减到 50 行 |
| **工具描述模糊** | 增加触发/禁止场景 | 添加 `【触发条件】` `【禁止场景】` |
| **冗余中间层** | 简化路由，直连工具 | intent_router → DeerFlow Tool |
| **输出格式要求过高** | 降低格式要求 | 允许 Markdown + 简化 JSON |
| **缺少追踪日志** | 增加关键路径日志 | 添加 `logger.info(f"[模块] 关键信息")` |
| **缺少纠错机制** | Prompt 增加纠错规则 | "用户纠错时，立即停止并重新理解" |

## 五、适配模型能力梯度

### 模型能力分级

| 等级 | 模型示例 | 核心能力 | Prompt 设计策略 |
|------|---------|---------|----------------|
| **L5 顶级** | Claude 4、GPT-4 | 强推理、强指令遵循、强纠错 | 可用复杂 Prompt、严格 JSON |
| **L4 高级** | Claude 3.5、GPT-3.5 | 较好推理、较好遵循 | 中等复杂度、放宽 JSON |
| **L3 中级** | glm-4、DeepSeek | 一般推理、需明确指令 | 精简 Prompt、简化输出 |
| **L2 初级** | glm-5、国内中小模型 | 需极简指令、易受干扰 | 极简规则、纯文本输出 |
| **L1 基础** | 小参数模型 | 只能执行简单任务 | 单一规则、无复杂约束 |

### 适配原则

```
模型能力 ↓ → Prompt 要求 ↓ → 输出约束 ↓

Claude：允许 500 行 Prompt + 严格 JSON
glm-5：限制 50 行 Prompt + Markdown 输出
```

## 六、最佳实践清单

### System Prompt 设计

1. **开头明确优先级**：`⚠️ 覆盖框架规则`
2. **只保留核心规则**：不超过 5 条核心原则
3. **用表格而非文字**：表格更易理解
4. **给出正反示例**：`✅ 正确做法` `❌ 错误做法`

### 工具定义设计

1. **必须有触发条件**：`【触发条件】用户说"X"时调用`
2. **必须有禁止场景**：`【禁止场景】用户说"Y"时 → 调用其他工具`
3. **参数说明简洁**：不超过 3 行
4. **返回值说明简洁**：只写关键字段

### 架构设计

1. **控制层级数量**：不超过 3 层处理链路
2. **单一真相来源**：规则/配置只在一处定义
3. **关键路径日志**：每个决策点都要有日志
4. **热加载机制**：配置修改无需重启

## 八、案例库（本次 Her 项目诊断）

### 案例：glm-5 + DeerFlow 表现"智障"

**现象**：工具调用错误、理解偏差、输出不稳定

**五问法追踪**：
```
问题现象：工具调用错误
├─ 为什么 1: 模型未能正确选择工具
├─ 为什么 2: 工具描述不够清晰，模型不知道何时调用
├─ 为什么 3: Prompt 存在多层冲突，模型不知道遵循哪层
├─ 为什么 4: 架构设计时没有统一规则优先级机制
└─ 为什么 5: 【根因】缺乏单一真相来源，规则分散在多层且存在冲突

根本对策：统一 Prompt 层级 + 精简内容 + 增强工具描述
```

**修复记录**：

| 修复项 | 修复前 | 修复后 |
|--------|-------|-------|
| SOUL.md | 590 行，多层规则冲突 | 100 行，明确覆盖框架规则 |
| 工具描述 | "查找匹配对象" | `【触发条件】用户说"找对象"时调用` |
| intents.yaml | 15 个意图，多层路由 | 7 个意图，直连 DeerFlow 工具 |
| 输出格式 | 严格 JSON | Markdown + 简化格式 |

## 九、持续改进机制

### 监控指标

| 指标 | 定义 | 预警阈值 |
|------|------|---------|
| **工具调用准确率** | 正确工具调用 / 总调用 | < 80% |
| **意图识别准确率** | 正确意图识别 / 总请求 | < 70% |
| **用户纠错率** | 用户纠错次数 / 总交互 | > 20% |
| **首次成功率** | 无纠错完成任务 / 总任务 | < 60% |

### 日志追踪标准

```python
logger.info(f"[模块名] 关键决策: intent=xxx, tool=xxx, confidence=xxx")
logger.info(f"[模块名] 执行结果: success=xxx, error=xxx")
logger.warning(f"[模块名] 异常情况: xxx")
```
