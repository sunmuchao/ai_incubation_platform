# Agent Native 架构核心原则

> **核心理念**：从"硬编码规则 + AI 调用"转向"AI 作为决策引擎 + 工具执行"

---

## 一、三层分离架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     System Prompt Layer                          │
│  所有决策逻辑通过 Prompt 表达                                     │
│  - 角色定义、决策规则、输出规范                                   │
│  - 不硬编码 if-else，用自然语言描述规则                           │
└─────────────────────────────────────────────────────────────────┘
                           ↓ 调用
┌─────────────────────────────────────────────────────────────────┐
│                     Tools Layer (Pure Execution)                 │
│  只做数据查询和执行，不含业务逻辑                                  │
│  - 返回原始数据（JSON），不做解读                                  │
│  - Agent 自己解读数据并决策下一步                                 │
└─────────────────────────────────────────────────────────────────┘
                           ↓ 依赖
┌─────────────────────────────────────────────────────────────────┐
│                     Data Layer                                   │
│  数据存储，无业务逻辑                                             │
└─────────────────────────────────────────────────────────────────┘
```

---

## 二、核心对比

| 维度 | 传统设计 | Agent Native 设计 |
|------|----------|-------------------|
| 规则表达 | 代码硬编码 | Prompt 表达 |
| 决策主体 | 规则引擎 | LLM Agent |
| 工具职责 | 包含业务逻辑 | 纯数据查询/执行 |
| 输出方式 | 预设模板 | AI 动态生成 |

---

## 三、执行循环

```
用户消息 → Agent 思考(LLM) → 选择工具 → 执行工具(纯数据) → 返回数据
                                                        ↓
                         Agent 再思考(LLM) ← 解读数据决定下一步
                                                        ↓
                         继续执行 或 输出回复 → 完成
```

---

## 四、开发规范

### 禁止

- ❌ Tool 中包含 if-else 业务判断
- ❌ 使用预设模板列表直接返回
- ❌ 代码硬编码"当 X 时输出 Y"

### 允许保留规则

- ✅ 商业限制（付费、次数）
- ✅ 安全边界（黑名单、封禁）
- ✅ 数据校验（必填、格式）

## 五、Agent "智障"诊断方法论

> **一句话总结**：Agent "智障" ≠ 模型不行，= **设计理念偏差**（把 Agent 当执行器） + Prompt 层级混乱 + 工具职责越界。
>
> **修复公式**：移除硬编码 → 简化工具返回 → 原始数据架构 → 升级适配模型。
>
> **核心原则**：Agent 是决策大脑，不是规则执行器。

### 5.1 四种反模式识别

| 反模式 | 表现 | 问题 | 修复 |
|--------|------|------|------|
| **触发词映射表** | 关键词 → 工具的硬编码表 | Agent 无法理解上下文 | 用自然语言描述角色和职责 |
| **工具返回预加工** | 返回 instruction + output_hint | 输出模板化、千篇一律 | 只返回原始 data |
| **工具包含业务逻辑** | 筛选、置信度计算在代码中 | 决策权被工具剥夺 | 业务逻辑移到 Prompt |
| **意图预分类层** | 先分类意图再路由 | Agent 只执行预设流程 | 移除预分类，Agent 自主决策 |

### 5.2 工具返回值处理规则

| 字段类型 | 用途 | Agent 应该做什么 |
|----------|------|-----------------|
| `instruction` | 给 Agent 的执行指令 | **执行**而非**输出** |
| `output_hint` | 给用户的回复提示 | **参考**而非**照抄** |
| `data` | 结构化数据 | **用于决策**而非**展示原始 JSON** |

**核心原则**：
```
instruction → 执行而非输出
output_hint → 参考而非照抄
data → 用于决策而非展示原始 JSON

永远不要把工具返回的 instruction 直接说给用户！
永远不要说"获取目标用户 XXX 的画像"这类内部状态描述！
```

### 5.3 模型能力适配梯度

| 模型等级 | Prompt 复杂度上限 | 输出约束 | 适用场景 |
|---------|------------------|---------|---------|
| Claude Opus/Sonnet | 500 行 OK | 严格 JSON | 复杂推理、多步决策 |
| Claude Haiku | 150 行 | Markdown | 简单意图、快速响应 |
| glm-4/DeepSeek | 100 行 | Markdown | 精简场景 |
| glm-5/中小模型 | 50 行 | 纯文本 | 极简场景 |

**适配策略**：Prompt 超出模型能力 → Agent 表现下降。优先升级模型，而非降低 Prompt 质量。

### 5.4 五问法根因分析

当 Agent 表现不如预期时：
```
问题现象：Agent 返回错误结果 / 不够智能
├─ 为什么 1: Agent 的决策是否被过度约束？
│   → 检查：是否有触发词映射表、大量 if-else
├─ 为什么 2: 工具设计是否违反 Agent Native 原则？
│   → 检查：工具返回是否包含 instruction/output_hint
├─ 为什么 3: 是否存在"意图路由器"层？
│   → 检查：是否有预分类逻辑
├─ 为什么 4: 模型选择是否合适？
│   → 检查：Prompt 复杂度是否超出模型能力
└─ 为什么 5: 【根本原因】设计理念是否把 Agent 当成"规则执行器"？
```

### 5.5 诊断检查清单

**System Prompt 检查**：
- [ ] 是否用自然语言而非触发词映射表？
- [ ] 是否只保留 5 条核心原则？
- [ ] 是否有明确的"覆盖框架规则"声明？

**工具定义检查**：
- [ ] 是否只返回原始数据（无 instruction/output_hint）？
- [ ] 是否不含业务逻辑（筛选、置信度计算）？
- [ ] 是否有明确的触发/禁止场景？

**架构检查**：
- [ ] 是否没有意图预分类层？
- [ ] 是否不超过 3 层处理链路？
- [ ] 每个 Agent 决策点是否有日志？
