## 一、根本问题 (Pain Point)

### 1.1 用户痛点

| 痛点 | 描述 | 影响人群 |
|------|------|----------|
| **选择困难** | 社区团购商品种类繁多，用户难以判断哪些商品适合自己 | 所有消费者 |
| **价格不透明** | 无法判断当前价格是否最优，担心买贵 | 价格敏感用户 |
| **成团不确定性** | 不知道团购能否成功，等待过程焦虑 | 参团用户 |
| **履约信息缺失** | 下单后不知道何时到货，无法安排取货时间 | 所有消费者 |
| **优惠信息分散** | 优惠券、活动规则复杂，难以最大化享受优惠 | 所有消费者 |

### 1.2 团长痛点

| 痛点 | 描述 | 影响程度 |
|------|------|----------|
| **选品困难** | 不知道什么商品好卖，凭经验判断 | 极高 |
| **备货不准** | 备货多造成损耗，备货少损失订单 | 极高 |
| **定价困难** | 不知道如何定价既能成团又有利润 | 高 |
| **用户运营复杂** | 缺乏工具分析用户偏好，无法精准营销 | 高 |
| **佣金不透明** | 不清楚佣金计算规则，收入预期不稳定 | 中 |

### 1.3 行业痛点

| 痛点 | 描述 | 市场影响 |
|------|------|----------|
| **同质化竞争** | 各平台功能相似，陷入价格战 | 行业利润低 |
| **供应链效率低** | 中间环节多，损耗高达 20-30% | 成本高企 |
| **数字化程度低** | 团长端数字化能力弱，依赖人工 | 效率瓶颈 |
| **用户需求响应慢** | 从需求产生到满足周期长 | 体验差 |

## 五、优先级判断标准

### 5.1 RICE 评分模型

所有功能需求按以下公式评分：

```
RICE Score = (Reach × Impact × Confidence) / Effort

其中:
- Reach (覆盖范围): 影响用户数 (1-10 分)
- Impact (影响力): 对核心指标影响 (0.25-3 分)
- Confidence (置信度): 数据支撑程度 (50%-100%)
- Effort (投入成本): 人天 (1-10 分)
```

### 5.2 优先级判定矩阵

| RICE 分数 | 优先级 | 处理方式 |
|-----------|--------|----------|
| > 50 | P0 | 立即实施，本周内 |
| 20-50 | P1 | 本月内实施 |
| 10-20 | P2 | 本季度实施 |
| 5-10 | P3 | 视资源情况 |
| < 5 | Backlog | 重新评估或放弃 |

### 5.3 一票否决规则

以下情况直接降级或拒绝：

| 场景 | 处理 |
|------|------|
| 违反"不做清单"任一原则 | 直接拒绝 |
| 缺少数据验证的假设性功能 | 降至 P2 以下 |
| 技术复杂度超出团队能力 3 倍以上 | 降至 P3 或寻找替代方案 |
| 与当前战略方向不一致 | 放入 Backlog |

### 5.4 快速通道规则

以下情况可直接升至 P0：

| 场景 | 处理 |
|------|------|
| 生产环境 P0 级 Bug | 立即修复 |
| 安全漏洞 | 24 小时内修复 |
| 竞品已验证且差距明显 | 2 周内补齐 |
| 高价值用户需求 (TOP10 团长) | 加速评估 |

#### P1 动态定价引擎 (v1.1.0 - 已完成)

**功能列表**:
- ✅ 动态价格配置 API: `/api/dynamic-pricing/prices/*`
- ✅ 动态价格计算 API: `/api/dynamic-pricing/calculate/{product_id}/{community_id}`
- ✅ 定价策略管理 API: `/api/dynamic-pricing/strategies`
- ✅ 价格弹性测试 API: `/api/dynamic-pricing/tests/*`
- ✅ 竞品价格追踪 API: `/api/dynamic-pricing/competitor-prices/*`
- ✅ 价格历史追踪 API: `/api/dynamic-pricing/prices/{product_id}/history`
- ✅ 价格统计报表 API: `/api/dynamic-pricing/stats`

**技术实现**:
- 定价策略：成团概率定价、需求弹性定价、竞品跟随定价、时间段定价、库存压力定价
- 价格边界保护：min_price/max_price 双重限制
- 价格历史记录：完整记录每次价格调整的原因、幅度、触发源
- 弹性 A/B 测试：支持对照组/实验组设计，自动计算价格弹性系数
- 竞品价格追踪：支持多平台竞品价格抓取和对比

**核心算法**:
- **成团概率定价**: 概率低于阈值时自动降价刺激，降价幅度 = base_price × elasticity × (threshold - probability)
- **需求弹性定价**: 需求低于/高于阈值时分别降价/涨价
- **竞品跟随定价**: 目标价格 = competitor_price × match_percentage，确保最低利润率
- **时间段定价**: 高峰时段涨价 5%，低峰时段降价 5%
- **库存压力定价**: 库存高于 80% 时降价 10% 促销
- **价格弹性系数**: (需求变化百分比) / (价格变化百分比)，绝对值>1 表示弹性充足

**验收结果**:
- 动态价格创建：成功
- 定价策略创建：5 种策略全部成功
- 价格计算：场景 1(成团概率低) 降价 20%，场景 2(需求高) 涨价 10%，场景 3(竞品低价) 跟随降价
- 弹性测试：弹性系数 -1.99，解释为"弹性充足"
- 竞品追踪：成功记录 3 个平台价格
- 手动调价：成功记录价格历史

**核心指标**:
- 价格调整响应时间：<50ms
- 策略评估准确率：100%（基于规则）
- 弹性测试完成时间：<1 秒

#### P9 智能履约调度系统 (v1.0.0 - 已完成)

**功能列表**:
- ✅ 自提点管理 API: `/api/fulfillment-scheduling/pickup-points`
- ✅ 配送路线优化 API: `/api/fulfillment-scheduling/routes/optimize`
- ✅ 人流预测 API: `/api/fulfillment-scheduling/traffic-prediction/{pickup_point_id}`
- ✅ 最佳取货时段 API: `/api/fulfillment-scheduling/optimal-pickup-times/{pickup_point_id}`
- ✅ 时间窗口推荐 API: `/api/fulfillment-scheduling/time-windows/recommend`
- ✅ 综合调度 API: `/api/fulfillment-scheduling/schedule`
- ✅ 异常处理 API: `/api/fulfillment-scheduling/exceptions/*`
- ✅ 调度仪表盘 API: `/api/fulfillment-scheduling/dashboard/summary`

**技术实现**:
- 路径优化算法：Haversine 距离公式 + 最近邻算法 + 2-opt 优化
- 人流预测模型：时序分析 + 时段特征工程 + 工作日/周末模式识别
- 时间窗口推荐：基于人流预测 + 营业时间约束 + 置信度排序
- 异常处理：多类型异常分类（延迟/路线变更/自提点关闭/容量超限等）
- 数据持久化：自提点、路线、任务、人流预测、时间窗口推荐、异常记录

**核心算法**:
- **VRP 路径优化**: 使用最近邻算法生成初始路线，2-opt 算法优化，可降低配送成本 30%+
- **人流预测**: 基于历史数据 + 时段系数（早高峰/午高峰/晚高峰）+ 工作日/周末模式
- **拥挤等级判定**: 根据预测人流与容量比例，分为 low/normal/high/crowded 四级

**验收结果**:
- 路径优化测试：5 个自提点，优化后总距离显著降低，优化分数>0.7
- 人流预测：24 小时预测，识别高峰时段和最佳取货时段
- 时间窗口推荐：基于拥挤程度和置信度排序，返回最优 3-5 个窗口
- 异常处理：支持 7 种异常类型，完整记录解决方案和影响评估

**核心指标**:
- 路径优化分数：0.5 + 改进比例（相对于原始顺序）
- 人流预测置信度：基于历史数据量，范围 0.3-0.9
- 异常解决率：可追踪和统计
